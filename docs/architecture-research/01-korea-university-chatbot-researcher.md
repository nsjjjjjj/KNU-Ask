# 국내 대학 챗봇 조사와 KNU-Ask 적용 요구사항

> 역할: Korea University Chatbot Researcher  
> 조사 기준일: 2026-07-20  
> 대상 저장소: KNU-Ask (`KNU`는 이 저장소의 README에 따라 **강남대학교**를 뜻함)  
> 주의: 현재 저장소와 샘플 전화번호·URL은 시연용이며 강남대학교 공식 서비스가 아니다.

## 1. 결론 요약

국내 대학의 학생지원 챗봇은 대체로 두 단계로 발전한다.

1. **공개 정보 안내형**: 대표 홈페이지, 학사 가이드, 공지, FAQ를 자연어로 검색하고 답변 근거 링크를 제공한다. 익명 이용이 가능하고 입시·학사·장학·취업·대학생활 안내에 적합하다.
2. **인증·개인화형**: 포털/LMS/도서관 등과 연계하여 내 수강과목, 출결, 과제, 시험, 장학금, 등록금, 학적 등 개인별 정보를 제공한다. 강한 인증·권한·감사 통제가 필요하다.

KNU-Ask는 1단계의 좋은 골격을 이미 갖췄다. 공지 변경 감지, 구조화 메타데이터, 임베딩 검색, 원문 링크, 담당 부서 표시, 근거가 약할 때 답변하지 않는 정책이 그 기반이다. 그러나 실제 서비스 전환에는 다음 다섯 가지가 가장 시급하다.

- 최근 2개월 공지만이 아니라 **현행 학사제도·학사일정·규정·부서 디렉터리·FAQ까지 출처 계층을 확장**한다.
- 공지의 PDF/HWP/HWPX/이미지 첨부를 추출하고, 표·각주·개정일을 보존한다.
- “현재 적용되는 정보”를 안정적으로 고르기 위해 **유효기간, 적용대상, 캠퍼스/학부, 학부·대학원 구분, 상충 출처 우선순위**를 데이터 모델에 넣는다.
- 답변마다 문장 또는 항목 단위로 출처와 기준 시각을 표시하고, 불확실·충돌·개인화 질문은 사람/공식 시스템으로 넘긴다.
- 콘텐츠 소유 부서가 수정·승인하고 실패를 추적하는 **운영자 검수·피드백·품질 대시보드**를 마련한다.

초기 운영 범위는 익명 공개정보 안내로 제한하는 것이 바람직하다. 포털 개인화는 별도 2단계로 분리하고, 챗봇이 학적 변경·수강신청·납부 같은 행위를 직접 실행하지 않도록 해야 한다.

## 2. 조사 범위와 방법

- 저장소의 README, API, 데이터 모델, 검색·생성 로직, 크롤러, 프런트엔드 흐름, 샘플 데이터를 직접 검토했다.
- 국내 대학이 공개한 챗봇 안내 페이지와 학사 가이드를 중심으로 기능·채널·데이터·운영 패턴을 비교했다.
- 대학 공식 페이지를 우선 인용했다. 사례의 제품명이나 모델 버전은 설계의 본질이 아니며, 각 대학의 공지 당시 상태임에 유의해야 한다.

### 3.1 공통 서비스 설계 패턴

**공개형과 개인화형의 경계가 분명하다.** 대표 홈페이지 기반 서비스는 익명 지식 안내에 집중하고, “내 성적/출결/등록금” 같은 질문은 로그인한 학사 시스템과 연계한다. RAG 문서에서 개인 정보를 추론하거나 사용자 입력만으로 개인 상태를 판단해서는 안 된다.

**근거 링크는 필수 기능이다.** 최근 생성형 챗봇 사례는 공식 페이지를 답변 근거로 밝히는 것을 주요 기능으로 내세운다. 따라서 링크는 부가 카드가 아니라 답변 품질 계약의 일부다.

**대학 정보는 시간과 대상에 강하게 종속된다.** 같은 “수강신청”이라도 학년도·학기·학년·재학생/복학생·학부/대학원·전공에 따라 답이 달라진다. 검색 전에 이 슬롯을 파악하고, 빠졌으면 질문하거나 모든 조건을 명시해야 한다.

**챗봇은 서비스 네비게이터다.** 답을 설명하는 데서 끝나지 않고 신청 페이지, 증명 발급, 담당 부서 전화, 상담 채널로 연결해야 한다. 다만 실제 신청·변경·납부는 공식 트랜잭션 시스템에서 완료하게 한다.

**다국어와 모바일 접근이 중요하다.** 외국인 유학생 지원을 명시한 사례가 많다. 한국어 원문을 근거로 다국어 답변을 생성하더라도 날짜, 금액, 서류명, 링크는 번역 과정에서 변형되지 않게 별도 구조화 필드로 다뤄야 한다.

**시범운영과 사용자 피드백이 일반적이다.** 처음부터 모든 행정업무를 포괄하기보다 범위를 정하고 실패 질문을 수집해 FAQ·동의어·문서를 보강한다.

## 4. KNU-Ask 현재 상태 평가

### 4.1 이미 잘 갖춘 요소

- 공지 본문 정규화 후 SHA-256으로 변경을 감지하고, 변경분만 재구조화·재임베딩한다.
- `NoticeMetadata`가 학년도, 학기, 신청기간, 대상 학년/학과/캠퍼스, 신청방법, 필요서류, 부서, 키워드 등을 수용한다.
- 메타데이터 필터와 벡터·키워드 점수를 결합하고, 임계값 미만 결과를 억지로 사용하지 않는다.
- 생성 프롬프트는 제공된 공지만 사용하고 없는 날짜·전화·링크를 만들지 않도록 한다.
- 원문 공지와 담당 부서를 답변 UI에 보여 주는 구조가 있다.
- 삭제된 공지를 즉시 지우지 않고 보관 처리하여 이력 추적 가능성을 남긴다.
- `MOCK_AI`, `MOCK_CRAWLER`, SQLite fallback과 테스트가 있어 개발·시연 재현성이 좋다.

### 4.2 실제 도입을 막는 주요 간극

| 영역 | 현재 구현 | 위험/영향 |
|---|---|---|
| 데이터 범위 | 최근 2개월 대표 공지 중심 | 상시 규정, 졸업요건, 증명발급, 장학제도 등 공지가 없을 때도 유효한 정보를 놓침 |
| 첨부파일 | 이름·URL 필드는 있으나 통합 추출 미실행 | 핵심 일정·대상·서류가 HWP/PDF/이미지에만 있으면 틀리거나 빈 답변 |
| 출처 단위 | 매칭 공지 카드와 최상위 원문 URL | 답변의 어떤 문장이 어느 문서 구절에 근거하는지 확인하기 어려움 |
| 검색 구조 | 공지 한 건당 임베딩 한 개 | 긴 공지·복합 첨부에서 관련 구절 회수율 저하, 세밀한 인용 불가 |
| 시간 판정 | `notice_status`에 의존 | 재공고·연장·수정·상시 제도와 행사 종료를 구분하기 어려움 |
| 대상 판정 | 모델은 필드를 보유하나 실제 검색 필터는 일부만 사용 | 학부/대학원, 학적상태, 학년, 소속에 맞지 않는 답변 가능 |
| 부서 정보 | 샘플 더미 번호, 상위 검색 결과의 부서만 표시 | 잘못된 연락처 또는 서로 다른 근거의 부서를 단일 부서처럼 제시 |
| 대화 맥락 | 세션은 touch만 하고 질문 캐시는 정규화 질문 전역 공유 | “그럼 2학기는?” 같은 후속 질문을 이해하지 못하며 사용자 조건별 답이 섞일 수 있음 |
| 카테고리/FAQ UI | 선택하면 의도적으로 빈 결과를 표시 | 사용자에게 서비스 장애처럼 보이고 탐색 경로로 기능하지 않음 |
| 운영 | 프로세스 내 BackgroundTasks, 수동 재처리 API | 재시도·스케줄·실패 큐·승인·롤백·콘텐츠 소유자 업무흐름이 없음 |
| 개인화 | 인증·내부 시스템 연계 없음 | “내 졸업 가능 여부/장학 대상 여부”에 답할 근거가 없음 |
| 품질 측정 | 단위·API 테스트는 있으나 실질 평가셋 없음 | 실제 학생 질문에서 정답성·근거성·누락을 수치로 판단하기 어려움 |

특히 전역 질문 캐시 키가 질문 문자열만 기준이라면 시간 경과에 따른 공지 변경, 언어, 사용자 맥락, 검색 인덱스 버전을 반영하지 못한다. 캐시 키에 지식베이스 버전과 질의 조건을 포함하고, 공지 갱신 시 관련 캐시를 무효화해야 한다.

## 5. KNU-Ask 기능 요구사항

### 5.1 P0 — 공식 파일럿 전에 반드시 필요한 기능

1. **답변 범위 계약**
   - 초기 범위: 공개된 학부 학사·등록·장학·병무·학생지원 공지와 제도 안내.
   - 범위 밖: 개인 성적, 학적, 납부 여부, 민감 상담, 법적·의학적 판단, 실제 신청 실행.
   - 첫 화면과 답변에 “공식 원문 확인” 및 비공식/시범 여부를 분명히 표시한다.

2. **출처 계층과 권위 모델**
   - 우선순위 예시: 승인된 내부 API/DB > 현행 대학 규정·학사편람 > 중앙 행정부서 공지 > 단과대/학과 공지 > FAQ > 기타 안내.
   - 동일 사안이 충돌하면 최신 날짜만으로 자동 결정하지 않고, 문서 권위·개정/시행일·적용대상을 함께 비교한다.
   - 충돌이 해소되지 않으면 두 근거를 표시하고 담당 부서 확인을 요청한다.

3. **문서/청크 단위 근거**
   - 공지와 첨부를 제목·절·표·목록 기준으로 청크화한다.
   - 각 청크에 문서 ID, 원문 URL, 첨부 URL, 페이지/절, 게시일, 개정일, 시행일, 수집시각, 콘텐츠 해시를 저장한다.
   - 답변의 일정·서류·대상·연락처 항목마다 직접 근거 링크 또는 페이지를 연결한다.

4. **첨부파일 처리**
   - PDF, HWP, HWPX, DOCX, XLSX, 이미지의 텍스트와 표를 추출한다.
   - 파일형식·용량 제한, 악성코드 검사, 중복 해시 캐시, OCR 신뢰도, 실패 재시도, 사람 검수 큐를 둔다.
   - 추출 실패 문서는 검색 가능 상태로 표시하지 않거나 “첨부 확인 필요”로 제한한다.

5. **시간·대상 조건 처리**
   - `document_status`와 `application_status`를 분리한다. 공지는 현행이지만 특정 신청기간은 종료될 수 있다.
   - 학부/대학원, 학년도, 학기, 학년, 학적상태, 소속, 캠퍼스, 학생 유형(신입·편입·유학생 등)을 명시적으로 저장·필터링한다.
   - 중요한 조건이 없으면 한 번의 짧은 확인 질문을 하거나 조건별 답을 나눠 보여준다.

6. **안전한 답변 형식**
   - “결론 → 대상 → 기간/마감 → 할 일 → 필요서류 → 공식 링크 → 담당부서 → 주의” 순서를 유지한다.
   - 날짜에는 연도와 한국 표준시를 명시하고, 현재 시각 기준 `예정/진행 중/마감` 배지를 계산한다.
   - 근거 없음, 근거 약함, 충돌, 추출 실패를 서로 다른 사용자 메시지로 구분한다.

7. **운영자 검수**
   - 부서별 콘텐츠 소유자, 승인자, 기술 운영자를 지정한다.
   - 신규/변경 문서, 낮은 구조화 신뢰도, 출처 충돌, 반복 실패 질문을 검수함으로 보낸다.
   - 승인·수정·반려 이력과 이전 버전을 보존하고 즉시 롤백할 수 있게 한다.

### 5.2 P1 — 정식 공개 서비스에 필요한 기능

- **대화 맥락**: 이전 턴의 학년도·학기·소속을 안전하게 유지하되 답변에 적용한 조건을 노출한다.
- **탐색형 UI**: 학사일정, 오늘 마감, 이번 주 신청, 카테고리별 최신 공지를 실제 데이터로 제공한다.
- **다국어**: 우선 영어를 도입하고 수요에 따라 중국어·베트남어 등을 추가한다. 중요 고유명사·날짜·금액의 번역 검증 세트를 운영한다.
- **상담 연결**: 정확한 담당 업무·전화·운영시간·문의 폼을 제공하며, 사용자가 대화 내용을 직접 선택해 전달하도록 한다.
- **피드백**: 도움이 됨/안 됨, 이유(틀린 정보·오래된 정보·출처 부족·질문 미이해·범위 밖), 자유의견을 수집한다.
- **접근성**: 키보드, 스크린리더, 고대비, 확대, 모바일 화면, 쉬운 한국어를 지원한다.
- **알림/추천은 별도 동의**: 마감 알림은 개인화 단계에서 명시적 선택을 받아 제공하고 언제든 해제 가능하게 한다.

### 5.3 P2 — 인증 후 개인화 단계

- 대학 SSO/OIDC로 인증하고 학사·LMS·도서관 API는 최소 권한의 읽기 전용 도구로 연계한다.
- 개인 정보는 벡터 DB나 일반 문서 인덱스에 적재하지 않고, 요청 시 시스템 오브 레코드에서 조회한다.
- 답변 생성 모델에는 필요한 최소 필드만 전달하고, 주민등록번호·여권번호·건강·상담 정보는 차단한다.
- “내 출결/시험/등록금”은 정보 조회로 시작하되 수강취소, 학적변경, 납부 등 상태 변경은 공식 화면으로 이동시킨다.
- 인증된 응답은 공유 캐시에 넣지 않고 사용자·권한·조회 시각이 분리된 짧은 캐시 또는 무캐시를 사용한다.

## 7. 검색·답변에 관한 도메인 권고

### 7.1 질의 분류

학생 질문은 다음 유형을 먼저 분류해야 한다.

- 제도 설명: “휴학은 몇 번까지 가능해?”
- 일정/마감: “2학기 복학 신청 언제까지야?”
- 절차/서류: “복수전공 포기하려면 뭘 내야 해?”
- 자격/대상: “편입생도 장학금 신청돼?”
- 위치/연락처: “수강신청 담당 부서 어디야?”
- 공지 탐색: “이번 주 마감하는 공지 알려줘.”
- 개인 상태: “내가 졸업 가능한가?”
- 트랜잭션: “휴학 신청해 줘.”
- 민감 상담/긴급 상황: 심리·건강·폭력·개인정보 등

마지막 세 유형은 일반 문서 RAG만으로 처리하면 안 된다. 개인 상태는 인증 시스템 조회로, 트랜잭션은 공식 화면으로, 긴급·민감 상담은 전문 채널로 전환한다.

### 7.2 검색 순서

1. 의도와 중요 슬롯을 추출한다.
2. 학부/대학원, 문서 유형, 적용기간, 대상, 권위로 후보를 필터링한다.
3. 키워드/BM25와 벡터 검색을 병렬 수행하고 재순위화한다.
4. 중복·정정·폐지·상충 관계를 해소한다.
5. 질문의 각 답변 항목을 직접 지지하는 청크만 생성 모델에 제공한다.
6. 충분성 검사 후 답변하거나 확인 질문/담당 부서 연결로 종료한다.

단순 유사도 임계값 하나보다 “요청된 핵심 필드(대상·기간·방법·서류)가 근거에 모두 있는가”를 평가하는 편이 대학 행정 질문에는 더 적합하다.

### 7.3 시간 관련 규칙

- “오늘/이번 주/다음 학기”는 서버의 Asia/Seoul 기준 절대 날짜로 변환한다.
- 게시일, 신청일, 행사일, 시행일을 혼동하지 않는다.
- 연장 공지는 원 공지와 연결하고 연장된 마감을 최종값으로 표시한다.
- 마감 시각이 없으면 임의로 18:00을 만들지 말고 “원문에 시각 미기재”라고 표시한다.
- 과거 공지는 역사 질문에만 사용하거나, 최신 공지가 없음을 명시한 참고 근거로만 사용한다.

### 8.1 소유와 책임

| 역할 | 책임 |

| 지식 운영자 | 수집 실패, 첨부 추출, 메타데이터, 인덱스 상태 관리 |

| 기술 운영자 | API/DB/워커/모델/관측·장애 대응 |
| 개인정보·보안 담당 | 개인정보 흐름, 위탁·보유기간, 접근권한, 감사 검토 |

### 8.2 권장 갱신 SLA

- 긴급/정정 공지: 게시 후 5분 이내 감지, 15분 이내 검색 반영.
- 일반 공지: 30분 이내 반영.
- 연락처·업무분장: 매일 변경 감지 + 분기별 소유 부서 확인.
- 규정·편람: 변경 감지 즉시 검수 큐, 승인 후 반영.
- 크롤링/추출 실패: 즉시 경보, 지수 백오프 재시도, 일정 횟수 이후 수동 검수.

실서비스는 FastAPI 프로세스 내 작업에서 워커·스케줄러·재시도 큐로 분리하고, 마지막 성공 수집 시각과 데이터 신선도를 사용자/운영자에게 보여 줘야 한다.

### 8.3 품질 지표

출시 판단은 단순 만족도보다 다음 지표 묶음으로 해야 한다.

- **근거 정확도**: 답변 주장 중 인용 근거가 실제로 지지하는 비율.
- **업무 정답률**: 부서 승인 골든셋에서 대상·기간·방법·서류·연락처가 맞는 비율.
- **최신성**: 원문 게시/수정부터 검색 반영까지 걸린 시간.
- **안전한 거절률**: 근거 부족·개인화·범위 밖 질문을 올바르게 넘긴 비율.
- **잘못된 확답률**: 근거 없거나 충돌하는데 단정한 비율. 최우선으로 낮춰야 한다.
- **해결률**: 추가 상담 없이 공식 링크/절차까지 도달한 비율.
- **상담 전환 품질**: 올바른 부서로 연결된 비율과 재전환율.
- **검색 무결과율** 및 상위 실패 의도.
- **사용자 피드백**, 언어·기기·장애 접근성 그룹별 차이.

평가셋은 실제 학생 질문을 개인정보 제거 후 의도·표현·오타·은어·후속 질문별로 구성한다. 학기 전환, 공지 정정, 마감 연장, 학부/대학원 동명 제도, 서로 다른 학번 규정 같은 어려운 사례를 반드시 포함한다.

## 9. 단계별 도입안

### 0단계 — 데이터 기반 정비 (2~4주)

- 서비스 범위·소유 부서·정답 출처 우선순위를 합의한다.
- 더미 연락처와 URL을 제거하고 공식 부서 디렉터리를 만든다.
- 학사일정·규정·편람·공지 소스 목록과 이용 허가를 확정한다.
- 200~300개 질문의 부서 승인 골든셋을 구축한다.

### 1단계 — 공개정보 파일럿 (4~8주)

- 학사·등록·장학 3개 영역, 한국어, 공개 정보만 제공한다.
- 청크 인용, 첨부 추출, 충돌/검수 큐, 실제 카테고리 탐색을 구현한다.
- 학생·교직원 제한 파일럿으로 실패 질문과 피드백을 모은다.
- 잘못된 확답률, 근거 정확도, 신선성 기준을 충족한 뒤 공개한다.

### 2단계 — 학생지원 확대 (4~8주)

- 병무·취업·국제·기숙사·장애학생지원 등 승인된 공개 영역을 추가한다.
- 영어와 상담 인계를 도입한다.
- 모바일 대학 앱/대표 홈페이지에 동일 API로 제공한다.

### 3단계 — 인증 개인화 (별도 사업)

- SSO와 읽기 전용 학사/LMS API부터 시작한다.
- 개인정보 영향, 위탁, 접근권한, 감사로그, 모델 데이터 처리 조건을 별도 승인한다.
- 조회형 기능의 정확성과 권한 격리를 검증한 뒤 제한적으로 확대한다.

## 10. 최종 권고

KNU-Ask를 “공지에 답하는 챗봇”에서 “강남대학교의 공식 학생지원 정보 관문”으로 발전시키되, 생성 모델을 진실의 원천으로 두지 말아야 한다. 공식 문서와 내부 시스템이 원천이고, 챗봇은 질문의 조건을 해석해 적절한 근거·절차·담당 부서로 안내하는 인터페이스여야 한다.

가장 효과적인 다음 투자는 모델 교체가 아니라 다음 순서다.

1. 공식 데이터 소유권과 출처 우선순위 합의
2. 학사일정·규정·편람·첨부파일을 포함한 지식 범위 확장
3. 청크 단위 인용과 시간·대상·정정 관계 모델링
4. 부서 검수·피드백·품질 평가 운영체계
5. 공개형의 안정성 입증 후 인증 개인화 분리 도입

이 순서를 따르면 현재 코드의 장점인 변경 감지, 하이브리드 검색, 근거 부족 시 거절을 유지하면서 국내 대학 서비스에서 검증된 출처성·개인화 경계·상담 연결·다국어·점진 출시 패턴을 현실적으로 흡수할 수 있다.
