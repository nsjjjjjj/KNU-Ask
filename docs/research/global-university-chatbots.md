# 해외 대학 학생지원·행정안내 챗봇 사례 조사

- 조사 기준일: 2026-07-20
- 조사 범위: 미국, 영국, 유럽, 아시아, 호주
- 조사 대상: 입학, 등록·재정지원, 재학생 행정지원, 캠퍼스 생활, 생성형 AI 기반 서비스
- 비교 기준: 현재 강냉이 에스크 MVP(공지 기반 RAG, 웹 채팅, 비로그인, 상담원·학사시스템 미연동)

## 1. 조사 방법과 해석 기준

대학 공식 페이지, 공식 보도자료, 개인정보 처리 안내를 우선 사용하고, 기술 구조나 성과가 대학 자료에 없는 경우에만 공급사 사례 연구와 전문 매체를 보조 자료로 사용했다. 공개 자료에 없는 기능은 유추하지 않고 **미공개**로 표시했다.

이 문서에서 용어는 다음과 같이 구분한다.

- **FAQ 방식**: 사전에 작성한 질문·답변 또는 의도(intent)에 연결된 고정 답변을 제공하는 방식
- **검색 방식**: 키워드 검색, 의도 분류, 지식베이스 검색, 벡터 검색/RAG, 실시간 업무시스템 API 조회 중 확인된 방식
- **개인화**: 로그인한 사용자의 신분·과정·일정·상태에 따라 답변이나 알림이 달라지는 경우
- **학생 시스템 연동**: 시간표, 학생기록, LMS, 도서관, 카드 등 대학 업무시스템에서 개인 데이터를 조회하는 경우
- **사람 상담원 연결**: 같은 대화에서 상담원에게 실시간 전환하거나, 대화 맥락을 포함한 서비스 요청/티켓을 생성할 수 있는 경우. 단순 전화번호 표시는 별도로 구분
- **생성형 AI/RAG**: LLM 사용이 확인되면 생성형 AI로 분류하되, 대학 데이터 검색 후 답변을 생성한다는 근거가 없으면 RAG로 단정하지 않음

공개 페이지가 남아 있어도 현재 운영 여부가 명확하지 않은 오래된 사례가 있으므로, 각 사례에 **근거 시점**을 표시했다.

유형별로 보면 Pounce와 Lola가 입학·등록, Staffordshire Beacon과 Helsinki Hubot이 재학생 행정지원, Pounce가 재정지원, Deakin Genie가 캠퍼스 생활, HKUST Library Chatbot이 생성형 AI를 대표한다. 공개 근거가 충분한 6개 사례 중 **RAG 구현을 명시한 운영 사례는 확인되지 않았다**. 따라서 “LLM 사용 = RAG”로 해석해서는 안 된다.

## 3. 상세 사례

### 3.1 Georgia State University — Pounce

- **국가**: 미국
- **서비스명**: Pounce
- **대표 유형**: 입학 지원, 등록 및 재정지원, 재학생 유지 지원
- **근거 시점**: 대학 공식 성과 페이지는 현재 공개 중이며, 최초 운영·성과는 2016년부터 축적
- **주요 목적**: 입학 허가 후 실제 등록하지 않는 `summer melt`를 줄이고, FAFSA, 재정보조 서류, 예방접종 기록, 배치시험, 수강등록, 기숙사 등 입학 전 할 일을 완료하도록 지원한다. 이후 재학생의 행정 절차와 학업 지속 지원으로 범위를 넓혔다.
- **지원 채널**: 스마트폰 **SMS 문자** 기반의 양방향 대화. 학생이 질문할 수 있을 뿐 아니라 대학이 마감과 미완료 업무를 먼저 알리는 선제형 채널이다.
- **데이터 출처**: 대학이 설계한 14개 주요 등록 장애요인, 입학·등록·재정지원 지식베이스, 학생별 메시지 캠페인과 설문 응답. 학생의 입학 단계·해야 할 일에 맞춘 메시지가 제공되지만, 공개 자료만으로 SIS에 실시간 직접 연결되는지 또는 명단·상태를 배치 반입하는지는 구분할 수 없다.
- **FAQ 방식**: 대학이 작성한 질문·답변을 AI 의도 분류와 지도학습으로 매칭한다. 새 표현이나 미처리 질문은 공급사/대학 운영자가 검토해 지식베이스를 개선한다.
- **검색 방식**: 자연어 의도 분류 후 검수된 답변 데이터베이스에서 응답하는 검색·분류형 방식. 웹 전체를 검색하거나 LLM이 자유 생성하는 RAG라는 근거는 없다.
- **개인화 여부**: **예**. 학생별 시점, 등록 과업, 관련 마감에 맞춘 알림을 보내고 의사결정 변경이나 설문 응답을 수집한다.
- **학생 시스템 연동 여부**: **부분 확인**. 개인화된 입학자 명단과 등록 과업 데이터 사용은 분명하나, 실시간 SIS/CRM API의 읽기·쓰기는 공개 자료에 명시되지 않았다. 대학 공식 설명은 Pounce와 별도의 새 학생 포털을 함께 운영했다고 밝힌다.
- **사람 상담원 연결 여부**: **예**. 자동 답변이 어려운 메시지를 대학 직원에게 라우팅한다. 초기 5만 건 이상 메시지 중 직원 개입이 필요했던 비율은 1% 미만이었다.
- **장점**:
  - 학생이 이미 쓰는 SMS에 들어가 별도 앱 설치와 로그인 마찰이 작다.
  - 질문을 기다리는 수동 FAQ가 아니라 마감 전 선제 알림과 넛지로 실제 행동을 유도한다.
  - 무작위 대조시험으로 등록 완료율을 측정했다. Georgia State는 첫해 20만 건 이상 답변과 summer melt 22% 감소를 보고했다.
  - 1세대 대학생과 Pell Grant 수혜 학생의 이용이 평균보다 높아, 정보 접근 격차 완화 가능성을 보였다.
- **문제점**:
  - SMS는 피싱과 구분하기 어렵고, 민감한 재정·입학 정보를 문자에 얼마나 노출할지 엄격히 제한해야 한다.
  - 개인화 품질이 학생 상태 데이터의 최신성과 정확성에 의존한다.
  - 문자 캠페인이 많아지면 알림 피로, 수신 거부, 통신 비용 문제가 생긴다.
  - 공급사 자료가 기술·성과 설명의 상당 부분을 차지하므로 장기 운영비와 벤더 종속을 별도로 평가해야 한다.
- **강냉이 에스크에 적용 가능한 기능**:
  1. 장학금, 수강신청, 등록금 납부, 졸업 신청 등 학생별 마감에 대한 opt-in SMS/카카오 알림
  2. “답변”보다 “해야 할 다음 행동”을 중심으로 한 체크리스트와 딥링크
  3. 미해결 질문을 담당 부서 티켓으로 자동 전환하고 대화·근거 공지를 함께 전달
  4. `질문 해결률`, `사람 전환율`뿐 아니라 신청·납부·제출 완료율을 KPI로 설정

**근거**: [Georgia State 공식 Summer Melt 성과](https://success.gsu.edu/reduction-of-summer-melt/), [Mainstay의 Pounce 운영·RCT 사례](https://mainstay.com/case-study/how-georgia-state-university-supports-every-student-with-personalized-text-messaging/), [Pounce 최초 운영 및 상담원 라우팅 설명](https://mainstay.com/blog/admithub-launches-first-college-chatbot-with-georgia-state/)

### 3.2 University of Staffordshire — Beacon

- **국가**: 영국
- **서비스명**: Beacon
- **대표 유형**: 재학생 행정지원, 캠퍼스 생활, 학업·복지 지원
- **근거 시점**: 2026년 현재 대학의 학생 서비스 및 개인정보 안내 페이지에 운영 기능이 게시됨
- **주요 목적**: 학생이 시간표, 교직원 연락처, PC 이용 가능 현황, 증명서, 뉴스·행사, 학생증 재발급, 동아리, 캠퍼스 길찾기, 복지·재정·국제학생 지원을 한 곳에서 이용하도록 한다.
- **지원 채널**: 웹 브라우저, iOS/Android 앱, Microsoft Teams 내 위젯. 로그인 후 텍스트 또는 음성 대화를 지원한다.
- **데이터 출처**: 약 400개로 시작한 서비스 FAQ와 대학 서비스 지식베이스, 시간표 시스템, 대학 메인 데이터베이스, 학생기록 시스템, 뉴스·이벤트, 사용자 유형·학습 장소·과정 수준·소속 그룹 프로필. 개인정보 안내에 연결 대상이 구체적으로 공개되어 있다.
- **FAQ 방식**: 부서별 검수 FAQ에 자연어 의도 인식과 머신러닝을 적용한다. 익명화·가명화한 실제 질문 일부를 직원이 감독 학습 방식으로 검토해 오인식을 교정한다.
- **검색 방식**: 클라우드 기반 NLU로 의도를 분류하고 관련 지식베이스를 조회한다. 시간표, 개인 튜터, 대출 관련 정보 같은 개인 질문은 대학 업무시스템에서 실시간성 데이터를 가져온다. 생성형 LLM/RAG 사용은 공개 자료에서 확인되지 않는다.
- **개인화 여부**: **예**. 사용자명과 요청을 연결하고 학생 유형, 캠퍼스/학습 장소, 과정 수준, 그룹에 따라 대시보드·뉴스·접근 가능한 지식베이스를 달리한다.
- **학생 시스템 연동 여부**: **예**. 시간표, 메인 대학 DB, 학생기록, 개인 튜터·강사, 가상 학생증, 학생 대출 추적 정보를 조회한다.
- **사람 상담원 연결 여부**: **부분적으로 예**. 개인 튜터와 강사의 연락처를 보여주고 지원 부서로 안내하며, 복지 이상 신호를 직원에게 알려 후속 연락하도록 설계되었다. 다만 현재 공개 페이지는 모든 챗봇 대화가 동일 창에서 실시간 상담원에게 전환된다고 명시하지는 않는다.
- **장점**:
  - 챗봇을 독립 FAQ 위젯으로 두지 않고 학생 앱·Teams·업무 기능을 묶은 디지털 허브로 설계했다.
  - 개인화 범위와 연동 데이터, 메시지 학습 방식을 개인정보 처리 안내에 투명하게 공개한다.
  - 단순 안내를 넘어 증명서 요청, 학생증 재발급, 시간표 확인 같은 실행 기능으로 이어진다.
  - 텍스트·음성, 모바일·웹·Teams를 지원해 접근 채널이 넓다.
- **문제점**:
  - 여러 민감 시스템을 한 앱에 연결하므로 계정 탈취 시 영향 범위가 크고 최소권한·감사로그가 필수다.
  - 복지 상태와 행동 데이터를 이용한 선제 개입은 유용하지만 감시로 느껴질 수 있어 동의, 설명, 거부권이 필요하다.
  - 기능이 많아질수록 지식 소유 부서, 답변 승인, API 장애 시 대체 경로를 관리하기 어렵다.
- **강냉이 에스크에 적용 가능한 기능**:
  1. 학교 SSO 로그인 후 학적·캠퍼스·학년을 메타데이터 필터로 사용
  2. 읽기 전용 시간표, 학사일정, 담당 지도교수/부서 연락처 카드
  3. 웹 외에 대학 포털·Microsoft Teams/모바일 앱에 같은 챗봇을 임베드
  4. “증명서 발급”, “학생증 재발급”을 안전한 공식 페이지 딥링크로 연결
  5. 개인정보 처리방침에 챗봇이 읽는 시스템, 보존기간, 학습 이용 여부를 항목별 공개

**근거**: [Staffordshire 공식 Beacon 기능·채널](https://www.staffs.ac.uk/students/digital-services/beacon), [Beacon 개인정보 안내와 시스템 연동](https://www.staffs.ac.uk/legal/data-protection/beacon-privacy-statement), [Beacon 초기 AI 코치·400 FAQ 설명](https://www.staffs.ac.uk/news/2019/01/introducing-beacon-a-digital-friend-to-staffordshire-university-students)

### 3.3 Universidad de Murcia·Universidad Politécnica de Cartagena — Lola

- **국가**: 스페인
- **서비스명**: Lola
- **대표 유형**: 입학 지원, 사전등록·등록 안내
- **근거 시점**: 대학의 2018년 소개 자료와 공급사의 2022년 누적 운영 자료가 확인됨. 2026년 현재 계속 운영되는지는 별도 확인 필요
- **주요 목적**: Murcia 지역 단일 대학지원 체계의 예비 학생에게 입학시험, 지원, 사전등록, 학부 등록 절차를 24시간 안내하고, 입학 성수기의 University Information Service(SIU) 반복 문의를 줄인다.
- **지원 채널**: SIU 웹사이트와 예비 학생용 DURM 모바일 앱.
- **데이터 출처**: SIU가 축적한 입학·사전등록·등록 관련 수천 개 FAQ와 대학 절차 정보. 공급사는 약 270개 핵심 질문을 다양한 표현으로 이해하도록 운영했다고 설명한다.
- **FAQ 방식**: 검수된 FAQ를 중심으로 한 자연어 의도형 챗봇. 상호작용에서 새로운 질문 표현을 수집하고 운영자가 지식을 보완하는 지속 학습 방식이다.
- **검색 방식**: 질문을 의도로 분류해 사전 작성 답변을 반환한다. 공개 웹 검색, 개인 지원서 조회, LLM 생성 또는 RAG의 근거는 없다.
- **개인화 여부**: **낮음/미확인**. 예비 학생의 질문 맥락에는 대응하지만 개인의 실제 원서 상태에 따른 답변이라는 근거는 없다.
- **학생 시스템 연동 여부**: **미확인**. 등록 절차를 설명하지만 챗봇 안에서 지원 상태를 조회하거나 등록을 실행한다는 자료는 없다.
- **사람 상담원 연결 여부**: **명시적 전환 미확인**. SIU 직원이 복잡하거나 감정적 판단이 필요한 문의에 집중할 수 있게 한다는 운영 목적은 확인되지만, 동일 대화의 상담원 전환 기능은 공개 자료에 없다.
- **장점**:
  - 입학·등록이라는 좁고 반복성이 높은 범위를 먼저 선택해 답변 정확도와 운영 효과를 높였다.
  - 예비 학생이 실제 사용하는 지역 공통 모바일 앱에 포함되어 발견성이 높다.
  - 공급사 집계 기준 24/7 약 270개 핵심 질문, 94.34% 정확도, 2019년 이후 59,137명·234,000개 메시지라는 운영 지표가 공개되었다.
- **문제점**:
  - 고정 FAQ는 규정 변경 반영이 늦으면 자신 있게 오래된 답을 줄 수 있다.
  - 정확도 산식, 실패 유형, 독립 검증 여부가 충분히 공개되지 않았다.
  - 개인 원서 상태 조회나 상담원 티켓 전환이 확인되지 않아 복잡한 사례의 해결이 끊길 수 있다.
  - 최근 운영 여부와 현대적 생성형 AI 전환 여부가 불명확하다.
- **강냉이 에스크에 적용 가능한 기능**:
  1. 전체 학사 업무를 한 번에 자동화하기보다 `수강신청`, `장학`, `졸업`처럼 고빈도 도메인별 책임자를 지정
  2. 같은 의미의 학생 표현을 실패 로그에서 수집해 FAQ·검색 평가셋에 지속 추가
  3. 도메인별 정답률, 미응답률, 오래된 답변 비율을 관리자 대시보드로 공개
  4. 모바일 대학 앱과 웹에서 동일 지식베이스를 사용하되 버전과 시행일을 답변에 표시

### 3.4 University of Helsinki — Hubot

- **주요 목적**: 지원자와 재학생에게 입학, 등록, 학업, 학위 수행 관련 기본 안내를 24/7 제공하고, 해결되지 않는 문제를 HelsinkiUni Help의 전문가 채팅 또는 서비스 요청으로 이어준다.
- **지원 채널**: 대학 입학서비스와 Studies Service 웹페이지 우측 채팅 위젯. 기능성 쿠키 동의가 필요하다.
- **데이터 출처**: 대학 입학·학생서비스 지침과 FAQ. 어떤 문서 저장소나 색인을 사용하며 갱신 주기가 무엇인지는 공개 자료에 없다.
- **FAQ 방식**: 3개 언어로 대학 서비스 FAQ를 안내한다. HelsinkiUni Help에는 별도의 AI 강화 콘텐츠 검색 기능도 있지만, 이 검색이 Hubot 답변 엔진과 동일하다는 근거는 없다.
- **검색 방식**: 챗봇의 구체적 검색 알고리즘은 **미공개**. 따라서 생성형 AI나 RAG로 분류하지 않는다.
- **개인화 여부**: **낮음**. 사용자가 시작한 페이지의 언어에 맞춰 핀란드어·스웨덴어·영어를 선택하지만, 학생 개인 학적에 따른 답변은 확인되지 않는다.
- **학생 시스템 연동 여부**: **미확인**. 개인 학적·시간표 조회보다 공식 지침 안내와 지원 요청 접수에 초점이 있다.
- **사람 상담원 연결 여부**: **예**. 봇이 답하지 못하면 채팅에서 HelsinkiUni Help에 연락 요청을 남길 수 있다. 업무시간에는 지원자용 전문가 채팅도 제공한다.
- **장점**:
  - 봇, 업무시간 실시간 전문가 채팅, 비동기 서비스 요청을 한 지원 체계로 연결한다.
  - 핀란드어·스웨덴어·영어 3개 언어를 제공해 국제학생 접근성을 높인다.
  - 복잡한 사안은 처음부터 서비스 요청을 권장해 봇의 한계를 명확히 한다.
- **문제점**:
  - 챗봇 기술, 지식 출처, 답변 근거 표시에 대한 공개 정보가 적다.
  - 쿠키 동의를 하지 않으면 위젯이 보이지 않아 접근성과 발견성이 떨어질 수 있다.
  - 학생 시스템 연동이 확인되지 않아 개인화된 “내 등록 상태” 해결에는 한계가 있다.
- **강냉이 에스크에 적용 가능한 기능**:
  1. 답변 실패 시 기존 질문·검색 근거·학생 연락처를 포함한 서비스 요청 생성
  2. 업무시간에는 실시간 상담, 시간 외에는 챗봇과 비동기 티켓으로 자동 전환
  3. 한국어 외에 영어 FAQ·공지 번역 레이어를 제공하되 원문 언어와 번역 여부를 표시
  4. 단순 전화번호 제시보다 부서별 큐, 예상 응답시간, 티켓 상태 조회를 제공

- **사람 상담원 연결 여부**: **별도 채널 안내**. 평일 09:00~17:00 도서관 직원 WhatsApp 번호를 제공하지만, 챗봇 대화 맥락이 자동 이전되는지는 명시되지 않는다.
- **장점**:
  - 아시아 대학의 실제 학생서비스 영역에서 LLM 기반 챗봇을 운영하며, 전통 FAQ보다 복합 표현과 대화 맥락을 다루기 쉽다.
  - 학생 개발자와 도서관 직원이 공동 개발하여 현장 요구와 학습 경험을 결합했다.
  - 모델명, 부정확 가능성, 사실 확인 책임, 민감정보 입력 금지, 직원 연락 방법을 화면에 명시한다.
  - 익명 설계로 초기 도입의 개인정보 위험을 줄였다.
- **문제점**:
  - 생성형 답변에 출처 링크가 없으면 이용자가 정확성과 최신성을 검증하기 어렵다.
  - 운영 페이지는 수집 정보가 보관·분석될 수 있다고 밝히는 반면 베타 소개는 개인정보를 저장하지 않는다고 설명한다. 데이터 범위·보존기간을 더 명확히 해야 한다.
  - 개인화와 도서관 시스템 연동이 없어 “내 대출 만기” 같은 고가치 질문은 해결하지 못한다.
  - RAG 또는 근거 제한이 공개되지 않아 환각 통제 수준을 판단하기 어렵다.
- **강냉이 에스크에 적용 가능한 기능**:
  1. 답변마다 공지 제목, 시행일, 원문 URL, 인용 구간을 표시하는 현재 RAG 강점을 유지
  2. 명시적 “도움됨/도움 안 됨”과 실패 사유(오래됨·관련 없음·이해 못함) 수집
  3. 비로그인 공개 안내와 로그인 개인화 영역을 분리해 최소정보 원칙 적용
  4. 모델명보다 중요한 지식 기준일, 출처 범위, 보존기간, 사람 상담 가능 시간을 채팅 시작 화면에 표시

- **장점**:
  - 질문 답변과 개인 업무 데이터를 결합해 “내일 일정”, “내 과제”, “내 카드 잔액”에 답한다.
  - 캠퍼스 길찾기, 기도실, 의사, 자전거 보관소 등 생활 질문까지 포함해 학생 여정 전체를 다룬다.
  - 2019년 피크에 하루 최대 12,000건 대화, 2020년 누적 60,000회 이상 다운로드와 연간 110,000건 대화를 보고할 만큼 사용 규모가 컸다.
  - 대학이 내부 구축해 여러 기존 시스템을 공통 대화 계층으로 묶었다.
- **문제점**:
  - 다수 레거시 시스템 연결은 API 변경, 장애 전파, 권한 관리, 데이터 불일치 문제를 만든다.
  - 고도 개인화는 데이터 최소수집·동의·프로파일링 설명이 없으면 프라이버시 우려가 크다.
  - 전용 앱은 웹/SMS보다 설치 장벽이 있고, 오래된 클라이언트 유지보수 비용이 발생한다.
  - 공식 기능 설명이 오래되어 현재 운영 범위와 성과의 최신 검증이 필요하다.
- **강냉이 에스크에 적용 가능한 기능**:
  1. “공지 검색”을 넘어 시간표·과제·도서 대출을 읽기 전용 도구 호출로 단계적 확장
  2. 학생 질문 의도를 `정적 규정`, `최신 공지`, `내 상태 조회`, `업무 실행`으로 분류해 서로 다른 검색·API 경로 사용
  3. 첫 단계에서는 실제 변경 대신 공식 시스템의 해당 화면으로 SSO 딥링크하고, 감사·권한 체계가 갖춰진 뒤 제한적 쓰기 기능 도입
  4. 국제학생, 신입생, 졸업예정자 등 생애주기별 홈 카드와 선제 안내

## 4. 국내 사례·강냉이 에스크와 다른 특징

아래 비교는 “모든 국내 대학 챗봇”을 일반화한 것이 아니라, **공개 FAQ·웹 위젯 중심의 국내 대학 챗봇 패턴과 현재 강냉이 에스크 MVP**를 비교 기준으로 삼는다. 국내 개별 대학도 로그인·메신저·학사 연동 수준이 다르므로 도입 전 별도 벤치마킹이 필요하다.

### 4.1 답변형에서 행동 유도형으로

강냉이 에스크는 공지를 찾아 근거 있는 답을 주는 데 강점이 있다. Pounce는 여기서 더 나아가 학생별 미완료 과업을 먼저 알려 실제 FAFSA·등록·서류 제출 완료를 유도한다. 해외 선도 사례의 KPI도 단순 문의 감소보다 **등록률, 과업 완료율, 재학 지속률**에 가깝다.

적용 시에는 “장학금 신청 방법은?”에 답하는 데서 끝내지 않고 다음을 제공해야 한다.

- 대상 여부 확인에 필요한 조건
- 학생별 마감일과 남은 단계
- 공식 신청 화면 딥링크
- 마감 전 opt-in 알림
- 완료 여부 피드백

### 4.2 웹사이트 부가기능에서 학생 디지털 허브로

국내 공개형 챗봇과 현재 MVP는 웹사이트 우측 위젯처럼 독립 채널인 경우가 많다. Beacon과 Genie는 시간표, 학생기록, 지도교수, 도서관, 카드, 캠퍼스 지도까지 묶은 **인증형 학생 허브**다. 챗봇은 별도 서비스가 아니라 여러 대학 시스템을 호출하는 공통 대화 UI다.

강냉이 에스크에는 한 번에 전체를 연결하기보다 다음 순서가 적합하다.

1. 공개 공지·FAQ RAG와 정확한 출처 표시
2. SSO 및 학적·캠퍼스·학년 기반 읽기 전용 개인화
3. 시간표·신청 상태·도서 대출 등 고가치 조회 API
4. 공식 업무 화면으로 SSO 딥링크
5. 충분한 권한·감사 체계 이후 제한적인 신청·변경 기능

### 4.3 수동 질문에서 선제형 멀티채널 지원으로

Pounce의 핵심은 AI 모델보다 **SMS라는 도달 채널과 선제 알림**이다. Beacon도 웹·앱·Teams·음성을 함께 제공한다. 국내 환경에서는 SMS만 복제하기보다 카카오 알림톡, 대학 앱 푸시, 포털, 이메일 중 학생이 동의한 채널을 조합하는 편이 현실적이다.

필수 통제는 다음과 같다.

- 수신 동의와 주제별 알림 설정
- 메시지에 민감정보를 직접 쓰지 않고 인증된 화면으로 유도
- 공식 발신자와 피싱 방지 문구 고정
- 빈도 상한과 야간 발송 제한
- 알림 클릭·업무 완료를 연결한 효과 측정

### 4.4 “담당 부서 전화번호”에서 맥락 보존형 상담 이관으로

현재 강냉이 에스크는 담당 부서와 전화번호를 보여주지만 상담원이 사용자의 앞선 질문과 검색 결과를 받지는 않는다. Helsinki Hubot과 Pounce는 해결되지 않은 질문을 서비스 요청이나 직원 큐로 넘긴다. 이 차이는 복잡한 민원에서 사용자가 같은 설명을 반복하지 않게 한다.

목표 흐름은 다음과 같다.

```text
학생 질문
  → 공지·FAQ 자동 답변
  → 해결 여부 확인
  → 미해결 시 부서·우선순위 자동 분류
  → 질문, 대화 요약, 인용 공지, 학생 동의 정보를 CRM/티켓에 전달
  → 상담원 답변을 같은 채널로 회신
  → 해결 결과를 지식 개선 후보로 등록
```

CRM을 도입하더라도 챗봇 원문 전체를 무조건 저장하지 말고, 학생 동의와 업무상 필요한 최소 정보만 전달해야 한다.

### 4.5 개인화와 개인정보 투명성을 동시에 강화

Beacon은 어떤 대학 시스템에서 어떤 개인 데이터를 가져오는지 개인정보 페이지에 상세히 적는다. HKUST는 반대로 익명 공개 챗봇을 선택하고 민감정보 입력 금지와 생성형 답변 오류 가능성을 전면 표시한다. 해외 사례의 공통 교훈은 “무조건 개인화”가 아니라 **서비스 가치에 맞는 개인정보 경계와 투명한 설명**이다.

강냉이 에스크는 화면과 기술 경계를 두 영역으로 나누는 것이 좋다.

- **공개 모드**: 공지·FAQ·부서 안내, 로그인 불필요, 대화 최소 보관
- **개인 모드**: SSO 후 내 일정·신청 상태 조회, 명시적 동의, 세부 접근기록·보존기간 적용

### 4.6 생성형 AI보다 지식 운영이 먼저

HKUST는 생성형 AI를 사용하지만 사실 확인 책임과 부정확 가능성을 명시한다. Lola, Pounce, Beacon은 비교적 좁은 FAQ·의도 모델과 검수된 답변으로 높은 통제력을 확보했다. 해외 사례는 최신 LLM을 쓰는 것 자체보다 **지식 소유자, 갱신 주기, 실패 검수, 사람 이관**이 운영 성패를 좌우함을 보여준다.

강냉이 에스크의 공지 기반 RAG는 출처 추적 측면에서 유리하다. 여기에 다음을 보완하면 된다.

- 공지의 시행일·폐기일·대체 공지 관계 관리
- 답변 문장별 출처와 원문 구간 표시
- 검색 실패와 상충 공지 탐지
- 부서별 콘텐츠 승인·재검토 기한
- 생성 답변 평가셋과 회귀 테스트
- 확신도가 낮거나 개인 판단이 필요한 사안의 자동 이관

### 4.7 성과 측정 방식

해외 사례 중 Pounce는 무작위 대조시험, Lola는 정확도·사용자·메시지 수, Genie는 다운로드·대화량을 공개했다. 국내 구축에서도 “챗봇 오픈” 자체가 성과가 되지 않도록 다음 KPI를 단계별로 두는 것이 중요하다.

| 단계 | 권장 KPI |
|---|---|
| 답변 품질 | 출처 적합률, 근거 충실도, 최신성 오류율, 상충 공지 탐지율 |
| 해결 품질 | 첫 답변 해결률, 재질문율, 사람 이관률, 티켓 재개방률 |
| 학생 행동 | 신청·제출·납부 완료율, 마감 누락률, 알림 수신 거부율 |
| 운영 효율 | 부서별 절감 문의시간, 평균 처리시간, 지식 갱신 소요시간 |
| 형평성 | 국제학생·장애학생·신입생 등 집단별 이용률과 해결률 격차 |
| 안전·신뢰 | 개인정보 사고, 잘못된 고위험 답변, 출처 클릭률, 사용자 신뢰도 |

## 5. 강냉이 에스크 권장 도입 우선순위

### P0 — 현재 MVP에서 바로 강화

1. 답변의 공지 시행일, 원문, 인용 구간, 담당 부서를 함께 표시한다.
2. “도움됨/안 됨”과 실패 이유를 수집하고 부서별 검수 큐를 만든다.
3. 해결되지 않은 질문을 이메일이 아닌 구조화된 티켓으로 만들고 대화 맥락을 보존한다.
4. 답변할 근거가 없거나 공지가 상충하면 생성하지 않고 상담으로 넘긴다.

### P1 — 인증·개인화

1. 학교 SSO를 연결하되 공개 질문은 계속 비로그인으로 허용한다.
2. 학적 상태, 캠퍼스, 학년을 검색 필터로 사용한다.
3. 시간표, 신청 상태, 장학·등록금 상태를 읽기 전용 API로 조회한다.
4. CRM/헬프데스크와 연동해 부서 배정, SLA, 처리 상태를 보여준다.

### P2 — 선제 알림과 멀티채널

1. 학생이 선택한 주제만 카카오 알림톡/SMS/앱 푸시로 보낸다.
2. 학생 상태에 맞는 마감·미완료 과업만 선별한다.
3. 메시지에서 민감정보를 노출하지 않고 인증된 포털 딥링크를 사용한다.
4. 대조군 또는 단계적 도입으로 실제 완료율 개선을 측정한다.

### P3 — 제한적 업무 실행

증명서 발급 신청, 상담 예약처럼 위험이 낮고 되돌리기 쉬운 업무부터 도구 호출을 허용한다. 수강 변경, 등록금, 장학 심사처럼 고위험 업무는 본인 재인증, 최종 확인, 감사로그, 사람 승인 체계가 갖춰질 때까지 공식 시스템으로 연결만 한다.

## 6. 결론

해외 사례의 가장 큰 차별점은 더 큰 언어모델이 아니라 **학생 여정과 운영 시스템의 연결**이다. Pounce는 SMS와 개인화 넛지로 등록 행동을 바꾸고, Beacon과 Genie는 챗봇을 학생 시스템의 공통 인터페이스로 만들었으며, Helsinki는 사람 상담과 티켓의 연속성을 확보했다. HKUST는 생성형 AI를 도입하면서 오류·개인정보 경계를 명시했고, Lola는 좁은 입학 도메인과 지속적인 FAQ 관리로 운영 효과를 냈다.

따라서 강냉이 에스크는 현재의 공지 기반 RAG와 출처 제시를 버리기보다 이를 신뢰 기반으로 삼아, **상담 이관 → SSO 읽기 전용 개인화 → CRM·학사시스템 연동 → opt-in 선제 알림** 순으로 확장하는 것이 적절하다. 생성형 AI 고도화보다 먼저 지식 책임자, 데이터 권한, 상담 운영, 성과 측정 체계를 설계해야 한다.
