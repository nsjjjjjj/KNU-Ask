# 강냉이 에스크 UX 및 학생지원 권고안

- 작성 기준일: 2026-07-20 (KST)
- 대상: 강남대학교 학생용 공지 기반 AI 문의 챗봇 MVP
- 우선 환경: 모바일 웹(최소 320px, 기준 검증 폭 375px)
- 조사 근거: [국내 대학 챗봇 조사](./korea-university-chatbots.md), [해외 대학 챗봇 조사](./global-university-chatbots.md)

## 1. 결론 요약

강냉이 에스크의 첫 화면은 긴 서비스 소개가 아니라 **즉시 질문할 수 있는 입력창과 시기별 추천 질문**을 먼저 보여줘야 한다. 자유 질문을 기본 경로로 두되, 질문을 어떻게 써야 할지 모르거나 용어를 모르는 학생을 위해 `문의 분야`와 `FAQ`를 보조 경로로 함께 제공한다.

답변은 채팅 문장만 길게 늘어놓지 않고 다음 순서로 구성한다.

1. 지금 학생에게 해당하는 결론
2. 학생이 해야 할 다음 행동과 기한
3. 담당 부서와 연락 방법
4. 근거 공지와 원문 링크
5. 추가 질문 또는 문의 종료

핵심 원칙은 **답을 보여주는 챗봇이 아니라, 공식 근거를 확인하고 다음 행동까지 끝낼 수 있게 하는 학생지원 흐름**이다.

## 2. 조사 결과에서 도출한 설계 원칙

국내 사례에서는 자유 질의와 함께 인기 질문·분야별 메뉴·시기별 퀵메뉴를 제공하는 혼합형 진입이 반복해서 나타났다. 숙명여대와 상명대 사례는 공식 문서 출처와 원문 링크가 생성형 답변의 신뢰를 높인다는 점을 보여준다. 서울사이버대와 가천대 사례에서는 AI가 해결하지 못할 때 전화·상담·담당 부서로 이어지는 종료 경로가 중요했다.

해외 사례의 차별점은 답변 자체보다 학생 여정과의 연결이다. Pounce는 해야 할 일을 안내해 실제 과업 완료를 유도하고, Helsinki Hubot은 미해결 문의를 사람에게 전달한다. Beacon과 Genie는 학생지원 기능을 여러 대학 시스템의 진입점으로 확장했다. 따라서 강냉이 에스크도 답변 길이나 대화량보다 **원문 확인률, 다음 행동 완료율, 미해결 문의의 안전한 이관율**을 우선해야 한다.

이를 다음 여섯 가지 원칙으로 정리한다.

- **질문 우선**: 첫 화면에서 설명보다 질문 시작을 빠르게 한다.
- **선택지는 보조**: 카테고리는 AI가 이해하기 위한 필수 단계가 아니라 학생의 탐색을 돕는 선택 사항이다.
- **결론 우선, 상세는 접기**: 모바일에서 핵심 답과 행동을 먼저 읽고 근거는 필요할 때 펼친다.
- **행동 중심**: 답변마다 가능한 경우 하나의 명확한 다음 행동을 제시한다.
- **근거와 한계 공개**: 출처, 게시일, 공지 상태, 검색 범위와 실패 이유를 숨기지 않는다.
- **사람에게 연결**: 근거가 없거나 개인 판단이 필요한 질문은 억지로 답하지 않고 담당 부서로 전환한다.

## 3. 핵심 질문에 대한 답

### 3.1 첫 화면에서 무엇을 가장 먼저 보여줘야 하는가?

가장 먼저 보여줄 것은 **“무엇을 도와드릴까요?”라는 짧은 안내와 바로 입력 가능한 질문창**이다. 그 아래에는 현재 학사 시기에 맞는 추천 질문 3~4개를 한 줄 또는 세로 목록으로 보여준다.

권장 우선순위는 다음과 같다.

1. 서비스명과 `문의 종료`가 있는 간결한 헤더
2. 한 문장의 역할·범위 안내
3. 질문 입력창과 전송 버튼
4. 현재 시기 추천 질문 3~4개
5. `문의 분야`와 `자주 묻는 질문` 보조 버튼
6. 개인정보 입력 금지와 공식 원문 재확인 안내

첫 화면의 긴 환영 문구, 전체 기능 설명, 7개 카테고리의 상시 펼침은 질문 시작을 늦추므로 피한다. 카테고리와 FAQ는 버튼을 누르면 바텀시트 또는 인라인 패널로 펼치는 점진적 공개가 적합하다.

추천 질문은 고정하지 않고 수강신청·등록·장학·시험·졸업 등 학사 시기에 따라 교체한다. 예시는 `장학금` 같은 단어가 아니라 `이번 학기 교내장학금 신청 기한이 언제야?`처럼 구체적인 문장으로 쓴다.

### 3.2 자유 질문과 카테고리 선택 중 무엇을 우선해야 하는가?

**자유 질문을 우선하고 카테고리 선택은 보조로 둔다.** 학생은 자신의 문제를 학교 행정 분류 체계에 맞춰 번역할 필요가 없어야 한다. 카테고리를 먼저 강제하면 장학·등록처럼 경계가 겹치는 질문에서 잘못된 선택과 이탈이 생긴다.

다만 자유 입력만 제공하면 질문 작성에 익숙하지 않은 학생과 정확한 행정 용어를 모르는 학생이 막힐 수 있다. 따라서 다음의 혼합형 구조를 사용한다.

- 입력창은 언제나 바로 사용할 수 있게 한다.
- `분야로 찾기`와 `FAQ 보기`는 입력창 가까이에 동등한 보조 버튼으로 둔다.
- 카테고리를 선택해도 즉시 막연한 답변을 생성하지 않는다. 관련 FAQ와 최신 공지를 먼저 보여주고, 선택값은 이후 자유 질문의 검색 필터로만 활용한다.
- 선택한 카테고리는 입력창 위에 삭제 가능한 필터 칩으로 표시한다. 언제든 다른 분야 질문으로 전환할 수 있어야 한다.

### 3.3 AI 답변은 얼마나 길어야 하는가?

기본 답변 본문은 **모바일 한 화면 안에서 핵심을 파악할 수 있는 250~400자, 3~6문장**을 목표로 한다. 공지 제목·담당 부서·출처 카드는 이 길이에 포함하지 않는다.

답변은 세 단계로 나눈다.

- **한줄 결론**: 1~2문장. 가능 여부, 대상, 핵심 기한을 먼저 말한다.
- **필수 설명**: 조건이나 예외를 2~4개의 짧은 항목으로 정리한다.
- **자세히 보기**: 배경 설명, 세부 조건, 여러 공지의 차이는 접힌 영역에서 제공한다.

마감일, 금액, 자격처럼 실수 비용이 큰 정보는 길게 설명하기보다 `확인 기준일`, `적용 대상`, `원문 확인 필요`를 명확히 표시한다. 확실한 근거가 없으면 답변을 늘려 추정하지 않는다.

### 3.4 공지 여러 개가 검색되면 어떻게 보여줘야 하는가?

공지 카드는 **관련도와 최신성에 따라 정렬해 최대 3개만 우선 노출**하고, 나머지는 `관련 공지 더보기 (N)`로 접는다. 모바일 가로 캐러셀은 일부 항목을 놓치기 쉽고 스크린리더·제스처 사용성이 낮으므로 세로 목록을 사용한다.

각 카드에는 다음 정보를 표시한다.

- 자료 유형 배지: `공지`, 향후 `학칙`, `FAQ`
- 공지 제목: 최대 2줄, 핵심어를 임의로 생략하지 않음
- 게시일과 적용 기간 또는 마감일
- 상태 배지: `진행 중`, `마감 임박`, `종료`, `보관됨`
- 이 답변에 사용된 이유를 나타내는 한 줄 요약
- `원문 보기` 명시적 버튼

여러 공지가 같은 내용을 뒷받침하면 AI 답변 아래에 근거 목록으로 묶는다. 내용이 충돌하면 하나로 섞지 말고 답변 위에 `공식 공지의 내용이 서로 다릅니다` 경고를 표시하고, 각각의 게시일·담당 부서를 함께 보여준다. 최신 공지를 우선 제안하되 학생이 담당 부서에 확인하도록 안내한다.

### 3.5 데이터가 없을 때 어떤 문구가 신뢰를 높이는가?

`관련 데이터가 없습니다`만 표시하면 시스템 장애인지, 검색 범위 밖인지, 실제로 공지가 없는지 구분할 수 없다. 신뢰를 높이는 문구는 **확인한 범위, 답하지 않는 이유, 복구 행동**을 함께 말해야 한다.

권장 기본 문구는 다음과 같다.

> 현재 검색 중인 최근 2개월의 공식 공지에서 이 질문에 답할 근거를 찾지 못했습니다. 내용을 추측해 안내하지 않겠습니다. 질문을 조금 더 구체적으로 바꾸거나 담당 부서에 확인해 주세요.

상태에 따라 문구와 행동을 구분한다.

| 상태 | 권장 문구 | 바로 제공할 행동 |
|---|---|---|
| 검색 결과 없음 | 공식 자료에서 답할 근거를 찾지 못했다고 설명 | `질문 고쳐 쓰기`, `담당 부서 보기` |
| 검색 범위 밖 | 검색 기간·포함 사이트 등 실제 범위를 명시 | `전체 공지 검색`, `부서 문의` |
| 시스템 오류 | 자료가 없는 것처럼 말하지 않고 일시적 오류임을 설명 | `다시 시도`, `FAQ 보기` |
| 질문이 모호함 | 이해하지 못한 부분을 특정해 한 번만 되묻기 | 구체적인 후속 질문 칩 |
| 개인 확인 필요 | 공개 공지만으로 개인 상태를 확인할 수 없다고 설명 | 공식 시스템 딥링크 또는 부서 문의 |

`없습니다`, `모르겠습니다`, `다시 질문하세요`처럼 책임 범위와 회복 경로가 없는 표현은 피한다.

### 3.6 담당 부서 안내를 어느 위치에 보여줘야 하는가?

담당 부서는 **해당 AI 답변 안에서 다음 행동 바로 아래, 근거 공지 목록보다 위**에 배치한다. 학생이 답변과 연락처의 관계를 놓치지 않게 별도 공통 메뉴가 아니라 문맥에 붙인 카드로 보여준다.

권장 정보 순서는 `담당 업무 → 부서명 → 전화번호 → 운영시간 → 문의 링크/위치`다. 전화번호는 `전화하기` 버튼으로 제공하고, 현재 시간이 운영시간 밖이면 `현재 운영시간이 아닙니다`와 다음 운영 시작 시각을 표시한다. 운영시간을 확인하지 못한 경우 임의 값 대신 `운영시간은 원문 또는 부서 페이지에서 확인해 주세요`라고 쓴다.

예외적으로 검색 근거가 없거나 AI가 답변할 수 없는 경우에는 담당 부서 카드가 결론 직후의 **주요 해결 수단**이 되므로 출처보다 먼저, 더 높은 강조 수준으로 표시한다.

### 3.7 학생이 해야 할 다음 행동을 어떻게 강조할 것인가?

답변 요약 직후에 **`다음에 할 일` 카드 하나**를 둔다. 한 화면에 여러 기본 버튼을 경쟁시키지 말고, 가장 가능성 높은 행동 하나만 채운 색상의 주요 버튼으로 표시한다.

다음 행동 문장은 `확인하세요`가 아니라 동사·대상·기한이 있는 형태로 쓴다.

- 좋은 예: `3월 8일 17:00까지 장학 시스템에서 신청서 제출`
- 나쁜 예: `기간 내 신청 바랍니다`

카드는 다음 정보를 포함한다.

- 행동 제목: `장학금 신청하기`, `수강신청 내역 확인하기`
- 마감: 절대 날짜와 시간, 필요하면 `D-3` 보조 표기
- 대상/조건: 학생에게 적용되는 핵심 조건 1개
- 주요 버튼: 공식 업무 화면으로 이동
- 보조 행동: `원문 먼저 보기` 또는 `담당 부서 문의`

마감 임박은 색상만으로 알리지 않고 `마감 임박 · D-3`처럼 텍스트와 아이콘을 함께 쓴다. AI가 실제 신청 완료를 확인할 수 없는 MVP에서는 버튼을 눌렀다는 이유로 `완료` 처리하지 않는다.

| 추가 질문 | 답변 아래 문맥형 후속 질문 2~3개와 자유 입력 | P0 |

| 사람 상담 이관 | 동의 후 질문·근거·요약을 구조화해 전달 | P1 |

## 5. 상태별 상세 설계

### 5.1 정상 답변

표시 순서는 다음과 같다.

1. `공식 공지 기준` 상태 라벨과 확인 기준일
2. 한줄 결론
3. 필수 조건 또는 절차
4. `다음에 할 일` 카드
5. 담당 부서 카드
6. 관련 공지 최대 3개
7. `도움이 되었나요?`와 추가 질문

AI 말풍선 하나 안에 모든 카드를 중첩하면 화면 폭이 좁아진다. 모바일에서는 AI 아이콘과 말풍선 이후의 행동·부서·공지를 같은 응답 그룹의 **전체 폭 카드**로 이어 붙이는 편이 읽기 쉽다.

### 5.2 종료되거나 보관된 공지가 근거인 경우

종료 공지를 현재 유효한 절차처럼 요약해서는 안 된다. 답변 최상단에 다음 경고를 표시한다.

> 종료된 공지입니다. 아래 내용은 과거 안내이며 현재 신청에 그대로 적용되지 않을 수 있습니다.

이후 흐름은 다음과 같다.

- 최신 대체 공지가 있으면 `최신 공지 보기`를 주요 행동으로 제공한다.
- 대체 공지가 없으면 과거 내용은 접힌 상태로 두고 담당 부서 확인을 우선한다.
- `종료` 상태는 주황/빨강 색상뿐 아니라 텍스트 배지와 경고 아이콘으로 함께 표시한다.
- 검색 결과 전체가 종료 공지뿐이면 AI가 현재 일정이나 자격을 추정하지 않는다.

### 5.3 결과 없음과 장애

검색 결과 없음, 네트워크 장애, 서버 오류를 모두 같은 `데이터 없음` 화면으로 합치지 않는다. 결과 없음은 검색 범위와 수정 방법을, 장애는 재시도와 대체 채널을 제공한다. 300ms 이상 기다리면 입력 전송 상태를, 1초 이상이면 답변 자리의 로딩 스켈레톤 또는 타이핑 상태를 보여준다.

### 5.4 추가 질문

답변 아래에 현재 문맥에서 자연스러운 후속 질문을 2~3개만 제안한다. 예를 들면 `신청 대상도 알려줘`, `필요 서류는 뭐야?`, `담당 부서에 전화할래`다. 칩은 최소 44px 높이와 8px 간격을 확보하고 가로 스크롤에만 의존하지 않도록 줄바꿈한다.

`다른 질문 입력`은 항상 가능해야 하며, 학생이 새 주제로 전환하면 이전 카테고리 필터를 유지할지 삭제할지 명확히 보여준다.

### 5.5 해결 여부와 문의 종료

답변 후 즉시 `종료하시겠습니까?`를 띄우면 후속 질문을 방해한다. 먼저 비차단형으로 묻는다.

> 이 답변으로 궁금한 점이 해결되었나요?

- `네, 해결됐어요` → 짧은 완료 피드백 후 `문의 종료` 또는 `새 질문하기`
- `아니요` → 실패 이유 선택(`내용이 달라요`, `정보가 오래됐어요`, `질문을 이해하지 못했어요`, `담당자 확인이 필요해요`)과 다음 경로 제공
- `추가 질문` → 같은 문맥으로 입력 계속

헤더의 `문의 종료`를 눌렀을 때 대화가 존재하면 확인 모달을 띄운다. 모달의 버튼은 `계속 문의하기`와 `문의 종료`처럼 행동 결과를 직접 적는다. 단순한 `예/아니오`는 피한다. 종료 시 대화가 초기화되고 복구할 수 없다면 이를 확인 문구에 명시한다.

## 7. 모바일 UI 및 접근성 기준

- 본문 글자는 최소 16px, 줄높이는 1.5 이상으로 한다.
- 버튼·칩·아이콘의 실제 터치 영역은 최소 44×44px, 인접 터치 영역 간격은 최소 8px로 한다.
- 색상만으로 진행·종료·오류 상태를 구분하지 않고 아이콘과 텍스트를 함께 사용한다.
- 일반 텍스트 대비는 WCAG AA 기준 4.5:1 이상을 유지한다.
- 화면 판독기 읽기 순서를 `결론 → 행동 → 부서 → 근거`의 시각 순서와 일치시킨다.
- 로딩·오류·새 답변은 `aria-live`로 알리되 긴 답변 전체가 반복 낭독되지 않게 상태만 먼저 알린다.
- 입력창에는 보이는 레이블 또는 지속적인 도움말을 제공하고 placeholder만 레이블로 사용하지 않는다.
- 하단 고정 입력창은 모바일 안전 영역을 반영하고 마지막 카드가 입력창 아래에 가려지지 않도록 여백을 확보한다.
- 긴 공지 목록은 세로 스크롤 하나만 사용하고 중첩 스크롤과 가로 스와이프를 피한다.
- 모달은 초기 포커스, 포커스 가두기, Escape/닫기, 닫은 뒤 원래 버튼으로 포커스 복귀를 지원한다.
- 동적 글자 확대와 `prefers-reduced-motion`에서도 정보가 잘리거나 기능이 사라지지 않아야 한다.

## 8. 현재 MVP에서 우선 수정할 항목

현재 화면은 자유 질문, 문의 분야, FAQ, 담당 부서, 공지 원문, 종료 모달의 기본 구조를 이미 갖추고 있다. 다음 항목을 먼저 개선하면 이번 권고 흐름에 가까워진다.

1. `관련 데이터가 없습니다`를 검색 범위·실패 이유·복구 버튼이 있는 상태별 안내로 교체한다.
2. 답변 본문과 부서/공지 카드 사이에 `다음에 할 일` 컴포넌트를 추가한다.
3. 공지 카드에 상태·게시일·적용 기간을 표시하고 종료 공지를 답변 상단에서 경고한다.
4. 관련 공지는 상위 3개만 먼저 보여주고 `더보기`를 제공한다.
5. FAQ와 카테고리 선택이 실제 검색/질문 전송으로 이어지게 하고, 선택한 분야를 필터 칩으로 보여준다.
6. 답변 아래에 해결 여부, 실패 이유, 문맥형 추가 질문을 배치한다.
7. 담당 부서의 전화번호를 `전화하기`로 만들고 현재 운영 여부를 구분한다.
8. 종료 모달의 `예/아니오`를 `문의 종료/계속 문의하기`로 바꾸고 키보드 포커스를 관리한다.

## 9. 검증 지표

출시 후에는 질문 수보다 학생이 실제로 해결했는지를 측정한다.

| 목표 | 핵심 지표 |
|---|---|
| 빠른 진입 | 첫 질문 전송률, 첫 질문까지 걸린 시간, 추천 질문 사용률 |
| 답변 이해 | 첫 답변 해결률, 같은 질문 재표현률, 상세 보기 사용률 |
| 신뢰 | 원문 클릭률, 종료 공지 오인율, 최신성 오류 신고율 |
| 행동 완료 | 주요 CTA 클릭률, 신청·조회 화면 도달률, 가능하면 업무 완료율 |
| 안전한 실패 | 결과 없음 후 복구율, 담당 부서 연결률, 반복 실패율 |
| 접근성 | 키보드·화면 판독기 과업 성공률, 375px 및 글자 확대 시 오류율 |

최소 사용성 검증은 신입생·재학생·졸업예정자·외국인 학생·보조기술 사용자 등 서로 다른 상황의 학생을 포함해 수행한다. 테스트 과업은 `장학 신청 기한 확인`, `종료된 공지 구분`, `자료가 없을 때 담당 부서 찾기`, `추가 질문 후 문의 종료`를 포함한다.
