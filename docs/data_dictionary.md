# 강냉이 에스크 수집·구조화 데이터 사전

이 문서는 전체 크롤링 전에 확정해야 하는 저장 기준이다. 공지에 없는 값은 추측하지 않고 `null`, 빈 배열 또는 빈 객체로 저장한다. 핵심 값의 근거가 불명확하면 `needs_review=true`로 남긴다.

## 1. 원문과 출처

- 원문 식별자, 제목, 원문 본문, 원문 URL
- 게시일, 수집일, 최종 처리일, 원문 내용 해시
- 출처 종류: 공식 공지, 상시 학사안내, 공식 FAQ, 담당자 안내, 행사안내
- 출처 우선순위, 보관 여부, 크롤링·첨부 추출 상태
- 본문에서 확인한 공식 링크와 신청 링크
- 구조화 스키마 버전, 임베딩 모델·버전

## 2. 첨부파일과 이미지

- 파일명, 원본 URL, SHA-256 해시
- MIME 형식, ETag, Last-Modified
- OCR·문서 추출 성공/부분 성공/실패 상태와 추출 문자 수
- 추출한 본문과 파일별 근거 표시
- 지원 형식: PDF, 이미지, HWPX, DOCX, PPTX, XLSX, TXT, CSV
- 구형 HWP처럼 자동 추출이 어려운 형식은 실패 상태와 검토 필요를 기록

## 3. 분류와 검색 조건

- 대분류, 세부 분류, 학년도, 학기
- 공지 행동 종류: 신청, 제출, 납부, 확인, 참석, 수강, 발급, 문의
- 실제 신청 가능 상태: 예정, 신청 중, 마감, 상시, 확인 필요
- 키워드, 동의어, 정규화된 검색 문장
- 대상 학생 상태, 학년, 학과, 캠퍼스와 자격·제외 조건

## 4. 모든 일정

- 학생이 실제 접수할 수 있는 신청·모집·제출·납부 시작/마감
- 교육·행사·근무·활동 시작/종료
- 면접, 서류 제출, 결제, 오리엔테이션, 결과 발표 등 복수 중요 일정
- 각 일정의 이름, 시작, 종료, 설명, 원문/첨부 근거 위치
- 게시일과 수행 기간을 신청 기간으로 사용하지 않음

## 5. 신청·업무 수행 정보

- 신청 방식, 신청 장소·온라인 경로, 검증된 신청 URL
- 순서가 보장된 1-2-3-4 단계, 단계별 행동과 링크
- 선행 조건, 준비 서류, 자격·제외 조건
- 비용·납부·환불 규정, 모집 인원
- 선발 방식·기준, 결과 발표 일정·방법
- 취소·변경·철회 조건
- 장학금·활동비·수료증 등 혜택
- 인정 학점·비교과 시간·봉사 시간·교육 시간
- 주의사항과 후속 확인 방법

## 6. 담당 부서와 문의처

- 담당 부서, 담당자 이름, 직책·역할
- 전화번호, 이메일
- 사무실 위치, 방문·운영 시간
- 공지별 문의처를 부서 대표 연락처보다 우선
- 신청자가 입력해야 하는 개인 연락처와 외부 기관 연락처를 담당자 정보로 오인하지 않음

## 7. 근거와 품질 관리

- 핵심 필드별 근거 위치: HTML 본문, 본문 이미지, 첨부 파일명·쪽수
- 문서·문단 임베딩과 검색에 사용한 정규화 문장
- 구조화 신뢰도, 단계별 신뢰도, 관리자 검토 필요 여부
- 첨부 추출 실패, 기간 없는 모집 공지, 지나치게 짧은 본문은 자동 검토 대상으로 표시
- 원문이 바뀌지 않으면 저장된 구조화·임베딩을 재사용해 AI를 다시 호출하지 않음

## 전체 수집 전 품질 게이트

1. 대표 질문 세트에서 올바른 원문이 1순위로 검색된다.
2. 신청 기간과 수행 기간이 분리된다.
3. 담당자·전화·이메일이 다른 기관 또는 신청자 정보와 섞이지 않는다.
4. 이미지·PDF 공지에서 신청 단계와 근거 위치가 보존된다.
5. 원문에 없는 단계·링크·기간을 생성하지 않는다.
6. 동일 원문 재처리 시 구조화·임베딩 호출이 발생하지 않는다.
7. 위 항목을 통과한 뒤에만 전체 크롤링과 전체 재인덱싱을 실행한다.

## 질문 중 자동으로 쌓는 데이터 갭

- 질문이 요구한 전화번호·담당자·이메일·기간·방법·링크·서류·자격·비용·인원·선발·결과·취소·혜택·학점/시간 필드가 선택된 공지에 없으면 자동 집계
- 검색 결과 없음, 신청 절차 근거 부족, 첨부 추출 실패, 낮은 구조화 신뢰도를 별도 유형으로 집계
- 같은 공지·필드·질문 의도는 한 항목으로 합치고 발생 횟수만 증가
- 부정확함·오래됨·근거 부족 등 사용자 피드백도 데이터 갭 대기열에 연결
- 개인정보 보호를 위해 질문 원문과 대화 원문은 저장하지 않고 분류·의도·요청 필드만 저장
- 관리자는 `/api/data-gaps`에서 발생 횟수순으로 확인하고 해결 또는 무시 상태로 관리
