## 총평

네 문서는 공개 자료와 추정을 구분하고, RAG를 무조건 만능으로 보지 않는다는 점에서는 신중하다. 그러나 최종 제안으로 갈수록 그 신중함이 약해진다. 아직 강남대학교의 실제 공지 분포, 학생 질문 로그, 정답 평가셋, 부서 운영 합의, 학교 인프라 표준, 개인정보 영향평가가 없는데도 하이브리드 검색, 검수 큐, 티켓, SSO, CRM, 멀티채널, 읽기 전용 학사 API까지 단계 계획에 포함한다. 이는 조사 자료가 보여 준 가능성을 이 프로젝트의 요구사항으로 바꾼 것이다.

현재 MVP의 가장 큰 위험은 기능 부족이 아니다. 최근 2개월치 공지와 미추출 첨부를 대상으로, 불완전하게 추출된 날짜·학기 메타데이터를 hard filter에 사용하면서도 사용자가 이를 학교의 공식 답처럼 받아들일 수 있다는 점이다. 공개 전 우선순위는 기능 확장이 아니라 검색 범위의 완전성, 날짜·대상 조건 회귀 테스트, 답변 거부 기준, 부서별 책임자, 개인정보 처리 경계의 검증이어야 한다.

## 1. 필수 검토 문서가 없는데도 보안·UX 결론을 전제로 함

- **주장**: 공개형/개인형 분리, 대화 최소 보관, 상담 이관, SSO, 멀티채널 같은 설계가 안전하고 학생 경험을 개선한다.
- **반론**: 지정된 UX 권고안과 보안 리뷰가 실제 저장소에 없으므로 이 주장을 독립적으로 검증할 수 없다. 특히 개인정보·접근성·인증·상담 이관은 부가 검토가 아니라 출시 가능 여부를 좌우하는 선행 조건이다. 누락 상태에서 아키텍처를 승인하면 “검토했다”는 잘못된 인상만 남는다.
- **근거**: `docs/research/ux-recommendations.md`와 `docs/reviews/security-review.md`가 존재하지 않는다. 반면 `global-university-chatbots.md` 4.4~4.5와 P1~P3, `infrastructure-options.md` 5장은 해당 주제의 결론을 이미 제안한다.
- **위험도**: **치명적**
- **수정 권고**: 두 문서를 작성·복구한 뒤 개인정보 흐름도, 위협 모델, 접근성 테스트, 학생 사용성 조사 결과를 이 리뷰에 재반영한다. 그전에는 SSO·티켓·개인화·멀티채널을 승인된 로드맵이 아니라 `검토 보류`로 표시한다.

## 2. 해외 성공 사례의 인과관계와 국내 전이 가능성을 과장함

- **주장**: Pounce식 선제 알림과 체크리스트를 적용하면 등록·신청·납부·제출 완료율을 높일 수 있다.
- **반론**: Pounce의 성과는 SMS 채널 하나의 효과가 아니라 대상자 선정, 학생 상태 데이터, 캠페인 운영, 상담 인력, 미국의 FAFSA·summer melt 맥락이 결합된 결과다. 이를 강남대학교의 재학생 공지 챗봇에 그대로 연결할 근거는 없다. 국내 학생의 알림 수용도, 카카오/SMS 동의율, 업무 완료 데이터 접근 가능성도 조사되지 않았다.
- **근거**: `global-university-chatbots.md` 3.1은 SIS 실시간 연동 여부조차 구분할 수 없다고 적지만, 4.1과 P2에서는 학생별 마감·미완료 과업과 완료율 개선을 권고한다. 성과 근거 일부는 공급사 사례 연구다.
- **위험도**: **높음**
- **수정 권고**: “적용 가능한 기능”을 “검증할 가설”로 낮춘다. 한 도메인, 한 채널, opt-in 집단으로 제한한 파일럿에서 도달률·차단률·업무 완료율을 측정하기 전에는 선제 알림을 로드맵에 확정하지 않는다.

## 3. 공개되지 않은 타 대학 아키텍처를 제품 근거로 사용함

- **주장**: 국내 사례는 공식 문서 기반 생성형 답변, 통합 검색, 개인화 같은 방향이 재사용 가치가 높음을 보여 준다.
- **반론**: 국내 조사 문서는 여러 대학의 임베딩, 벡터 DB, 검색 갱신 주기, 근거 선택 기준을 `확인 불가`로 올바르게 표시했다. 그런데 이 불명확한 사례를 강냉이 에스크의 RAG·검색 통합·개인화 방향을 정당화하는 근거로 재사용한다. 유사한 화면과 홍보 문구는 동일한 아키텍처나 운영 성과를 뜻하지 않는다.
- **근거**: `korea-university-chatbots.md`의 각 사례는 RAG를 대부분 `[추정]` 또는 `[확인 불가]`로 분류하고, 비교 분석에서도 실제 검색 아키텍처가 공통 확인되지 않는다고 명시한다. 그럼에도 결론은 “공식 문서 기반 생성형 답변” 조합을 최우선으로 제안한다.
- **위험도**: **높음**
- **수정 권고**: 타 대학 사례는 기능 아이디어와 위험 목록에만 사용한다. 강냉이 에스크의 아키텍처 선택 근거는 자체 코퍼스 평가, 비용 측정, 장애 시험, 사용자 연구로 한정한다.

## 4. 사례 선정 편향으로 실패 가능성을 낮게 보임

- **주장**: 해외 선도 사례는 학생 여정과 운영 시스템 연결이 핵심이며, 이를 단계적으로 확장하는 것이 적절하다.
- **반론**: 선정된 6개 사례는 공식 홍보 페이지가 남고 특징이 뚜렷한 서비스 위주다. 종료·축소·낮은 이용률·예산 중단·조직 갈등 사례가 거의 없다. 성공 사례 중심 표본으로부터 일반적인 도입 순서를 도출하면 생존자 편향이 생긴다.
- **근거**: `global-university-chatbots.md`는 Lola와 Genie의 최신 운영 범위를 재검증해야 한다고 쓰면서도 두 사례를 로드맵 근거로 사용한다. 국내 문서에서 명시적으로 종료가 확인된 사례는 가천대 한 건뿐이다.
- **위험도**: **중간**
- **수정 권고**: 종료·낮은 채택·정확도 논란·예산 초과 사례를 별도 표본으로 추가하고, 사례별 총소유비용·전담 인력·운영 기간·종료 이유가 확인되지 않으면 의사결정 가중치를 낮춘다.

## 5. 학생이 실제로 챗봇을 사용할 것이라는 근거가 없음

- **주장**: 웹 채팅, 시기별 퀵메뉴, FAQ, 멀티채널을 제공하면 학생의 정보 접근성이 개선된다.
- **반론**: 강남대학교 학생이 공지를 찾을 때 포털 검색, 학과 단체방, 에브리타임, 전화, 검색엔진 중 무엇을 우선하는지 조사되지 않았다. 챗봇은 질문을 문장으로 작성해야 하고 답을 다시 원문에서 확인해야 하므로, 단순 검색보다 단계가 늘 수 있다. 설치형 앱이나 별도 웹 위젯은 발견되지 않을 가능성도 높다.
- **근거**: 두 사례 조사에는 타 대학의 채널 설명은 있지만 강남대학교 학생 인터뷰, 검색 로그, 과업 테스트, 재방문율 자료가 없다. `README.md`의 현재 제품은 비로그인 웹 채팅 MVP다.
- **위험도**: **높음**
- **수정 권고**: 5~8개 핵심 과업으로 학생 사용성 테스트를 하고 기존 공지 검색과 성공률·소요시간을 비교한다. 첫 화면 노출 위치와 유입 경로가 확정되기 전에는 멀티채널 개발을 하지 않는다. 챗봇이 검색보다 느리면 검색형 UI를 기본으로 두고 대화는 보조로 둔다.

## 6. P0에 티켓·검수 큐를 넣은 것은 MVP 범위를 넘음

- **주장**: 현재 MVP에서 바로 구조화된 티켓, 대화 맥락 보존, 부서별 검수 큐를 구축해야 한다.
- **반론**: 티켓 시스템은 UI 하나가 아니라 부서 라우팅 규칙, 담당자 계정, 알림, SLA, 재배정, 중복 민원, 보존·삭제, 감사, 회신 채널을 요구한다. 부서가 실제로 큐를 처리하기로 합의하지 않았다면 챗봇이 학생의 기대만 높이고 문의를 방치한다.
- **근거**: `global-university-chatbots.md` P0 2~3번과 4.4는 검수 큐와 구조화된 티켓을 즉시 강화 항목으로 둔다. 현재 `README.md`는 상담원 연결과 관리자 검수 화면이 포함되지 않는다고 명시한다.
- **위험도**: **높음**
- **수정 권고**: MVP는 공식 문의 페이지·전화·이메일 딥링크와 사용자가 복사할 수 있는 질문 요약까지만 제공한다. 최소 1개 부서가 처리 책임과 SLA에 합의한 뒤 해당 부서에 한해 티켓 파일럿을 연다.

## 7. SSO·학사 API·CRM·알림·업무 실행은 독립 프로젝트에 가까움

- **주장**: 공개 RAG 이후 SSO 읽기 전용 개인화, CRM·학사시스템 연동, opt-in 알림, 제한적 업무 실행 순으로 확장하면 된다.
- **반론**: 이 기능들은 단순한 RAG의 다음 버전이 아니다. 인증 보증 수준, 권한 모델, 원장 시스템 책임, 동의 철회, 장애 시 정합성, 감사, 민원 처리, 학교 조직 간 계약이 필요한 별도 프로그램이다. 한 로드맵에 넣으면 MVP의 성공 기준과 예산이 흐려진다.
- **근거**: `global-university-chatbots.md` P1~P3와 결론은 위 확장을 권고한다. 같은 문서의 Beacon·Genie 문제점은 다중 시스템 연동과 권한 관리의 부담을 인정한다.
- **위험도**: **높음**
- **수정 권고**: MVP 범위를 공개 공지 검색과 공식 채널 연결로 고정한다. 개인화 이후 기능은 별도 개인정보 영향평가, 시스템 소유자 승인, 예산, 운영 인력이 확보될 때 새로운 프로젝트 제안서로 분리한다.

## 8. E 하이브리드 검색을 운영 기본안으로 정하기에는 평가 근거가 없음

- **주장**: 메타데이터 + PostgreSQL 전문 검색 + pgvector(E)가 실제 MVP 공개 전후의 운영 기본안이다.
- **반론**: 현재 공지가 수천 건 이하이고 정답 평가셋도 없다. 이 규모에서는 제목·본문 BM25 유사 검색이나 규칙 기반 필터가 충분할 수 있고, 벡터 검색이 오히려 숫자·학기·서식명 혼동을 늘릴 수 있다. D에서 E로 넘어갈 품질 개선 폭과 비용은 측정되지 않았다.
- **근거**: `rag-options.md` 9.3은 100~300개 평가 질문이 필요하다고 스스로 적지만, 1장과 14장은 그 평가 전부터 E를 목표로 확정한다. 현재 `HybridSearch`는 PostgreSQL 전문 검색이나 chunk 검색이 없는 애플리케이션 가중합이다.
- **위험도**: **높음**
- **수정 권고**: 키워드/전문 검색 단독(C), 현재 D, E 최소 버전을 동일 평가셋으로 비교한다. Recall@5뿐 아니라 명시 조건 위반율·최신성 오류율·무응답 정밀도를 측정해 통계적으로 유의한 개선이 있을 때만 E를 채택한다.

## 9. PostgreSQL 기본 전문 검색의 한국어 효과를 낙관함

- **주장**: PostgreSQL 하나로 전문 검색과 벡터 검색을 결합하면 이 프로젝트 규모에서 실용적인 하이브리드 검색이 된다.
- **반론**: PostgreSQL의 기능 지원과 한국어 검색 품질은 다른 문제다. 기본 파서는 조사·어미·복합명사·띄어쓰기 변형을 한국어 형태소 수준으로 해결하지 않는다. 정규화 토큰과 동의어 사전은 결국 지속적인 사전 운영을 요구한다. 별도 분석기를 “필요할 때” 도입하면 이미 색인·평가 체계를 다시 바꿔야 한다.
- **근거**: `rag-options.md` 3.E와 8장은 기본 설정의 한국어 한계를 인정하면서도 E의 정확도를 `매우 높음`으로 평가한다. 그 등급을 뒷받침하는 강남대학교 코퍼스 실험은 없다.
- **위험도**: **중간**
- **수정 권고**: “매우 높음” 등급을 삭제하고 미측정으로 표시한다. 학생 구어체, 띄어쓰기 오류, 학수번호, 서식명, 동음 표현을 포함한 한국어 평가셋으로 단순 ILIKE/trigram, FTS, 벡터, 조합을 먼저 비교한다.

## 10. 메타데이터 hard filter가 정답을 제거할 위험을 충분히 통제하지 못함

- **주장**: 사용자가 명시한 학년도·학기·학년·학생 상태는 메타데이터 hard filter로 제한한다.
- **반론**: hard filter는 추출값이 완전하고 정확할 때만 안전하다. 공지 한 건에 여러 학기·학년·캠퍼스가 포함되거나 제목과 첨부가 충돌할 수 있다. 현재 스키마의 학년도·학기는 단일 값이고, 현재 구현은 메타데이터 신뢰도를 필터 적용 여부에 반영하지 않는다.
- **근거**: `rag-options.md` 5.2와 12.1은 명시 조건 hard filter를 권고하는 한편 11장은 다중 값 가능성을 인정한다. `HybridSearch.search()`는 `academic_year`와 `semester`를 곧바로 SQL 조건으로 사용하고, `confidence`나 `needs_review`를 확인하지 않는다. `grade`와 `student_status`는 분석하지만 검색 조건에는 아예 적용하지 않는다.
- **위험도**: **치명적**
- **수정 권고**: 다중값 스키마와 필드별 신뢰도를 먼저 도입한다. 검수되지 않았거나 누락 가능한 필드는 후보 제거에 쓰지 말고 재순위화에만 사용한다. hard filter로 제거된 정답 비율을 별도 지표로 측정한다.

## 11. “이번 학기”와 학년도 해석이 실제 구현에서 틀릴 수 있음

- **주장**: 날짜·학기는 규칙 기반을 기준으로 하고 KST 및 대학 학사 일정으로 정확히 처리한다.
- **반론**: 문서의 권고는 맞지만 현재 제품은 이를 구현하지 않았다. 7월 전체를 1학기로 보는 단순 규칙은 하계 계절학기, 2학기 수강신청 기간, 학사 일정 경계에서 틀릴 수 있다. 2026학년도 동계 일정이 2027년에 실행되는 경우도 단일 달력 연도 필터와 충돌한다.
- **근거**: `rag-options.md` 7장은 “1~7월=1학기” 고정을 금지한다. 그러나 `AIService._mock_query()`는 `datetime.now().month <= 7`이면 이번 학기를 1학기로 정한다. 서버 시간도 명시적으로 `Asia/Seoul`을 사용하지 않는다.
- **위험도**: **치명적**
- **수정 권고**: 학교가 승인한 학사기간 테이블을 기준 데이터로 만들고 KST 기준 시각을 주입 가능하게 한다. 계절학기·수강신청 선행 기간·연도 경계를 포함한 테스트를 통과하기 전에는 “이번 학기”를 자동 확정하지 말고 사용자에게 기준 학기를 보여 주거나 되묻는다.

## 12. 최근 2개월 코퍼스로 현재·과거 질문을 함께 처리하려는 모순

- **주장**: 현재 신청 질문과 특정 과거 연도 질문을 구분하고, 종료·보관 공지도 의도에 따라 검색할 수 있다.
- **반론**: 원천 수집이 최근 2개월뿐이면 과거 공지와 학기 초 공지가 애초에 DB에 없다. “보관 공지 포함” 정책만으로 수집하지 않은 자료를 검색할 수 없다. 현재 유효한 연간 규정이나 학기 초 공지도 게시 후 2개월이 지나면 최신 크롤링 범위 밖이 된다.
- **근거**: `README.md`는 최근 2개월 공지만 수집한다고 명시한다. `rag-options.md` 10장은 특정 과거 연도 검색과 절차·서식 일반 안내를 지원하도록 권고한다. `korea-university-chatbots.md` 적용 제안은 범위 밖 질문에 제한을 표시하라고 할 뿐, 현재 질문에서 필요한 오래된 유효 공지 누락 문제를 해결하지 못한다.
- **위험도**: **치명적**
- **수정 권고**: 최초 전체 백필 범위와 지속 증분 수집 범위를 분리한다. 최소 1~2개 학년도와 상시 규정·학칙을 백필하고, 유효기간 기준으로 보존한다. 그전에는 과거 질의와 “현재 규정” 답변을 지원 범위 밖으로 명시한다.

## 13. 공지 상태를 저장값에 의존해 만료 답변을 낼 수 있음

- **주장**: 만료 상태를 현재 시각에 따라 재계산하고 질의 의도별로 감점한다.
- **반론**: 현재 구현은 처리 시 LLM/목업이 만든 `notice_status`를 저장하고 검색 시 그대로 점수화한다. 목업 구조화는 모든 공지를 `active`로 둔다. 실제 LLM 결과도 시간이 지나면 자동으로 만료되지 않는다. 문서가 제안한 안전장치가 존재하는 것처럼 발표하면 위험하다.
- **근거**: `NoticeProcessor.process()`가 구조화 결과의 상태를 저장하며, `HybridSearch.search()`는 저장된 값으로 점수를 더한다. 질의 시 기간 재계산 코드는 없다. `rag-options.md` 10장은 색인 시 한 번 정하고 끝내지 말라고 명시한다.
- **위험도**: **치명적**
- **수정 권고**: `application_start/end`, `event_start/end`, 상시 여부를 KST로 계산하는 결정적 함수를 구현하고 저장 상태는 캐시로만 취급한다. 날짜가 없거나 충돌하면 `unknown`으로 두고 확정형 표현을 금지한다.

## 14. 토큰 비용과 외부 API 비용을 지나치게 단순화함

- **주장**: 질문당 분석 0~1회, 임베딩 1회, 답변 1회이며 상위 소수 공지만 전달하므로 질의 비용은 낮다.
- **반론**: 비용은 호출 횟수뿐 아니라 입력 길이, 재시도, chunk 수, 첨부 OCR/비전, 평가·재색인, 실패 재호출, 상담 요약, 다국어 번역까지 포함해야 한다. 공지 변경 시 구조화 호출은 최대 3회 재시도하고, E에서는 chunk 수만큼 임베딩한다. 실제 학생 질문의 중복률이 없으므로 Redis 절감 효과도 미확인이다.
- **근거**: `rag-options.md` 4장은 비용을 상대 등급으로만 표시한다. `AIService.structure_notice()`는 최대 3회 호출하고, 답변은 최대 5개 매치 전체를 JSON으로 직렬화한다. `global-university-chatbots.md`는 향후 티켓 요약·다국어·알림까지 제안한다.
- **위험도**: **높음**
- **수정 권고**: 월 공지 변경량, 평균 원문/첨부 토큰, 질문 수, 캐시 적중률, 재시도율, 평가 호출량을 변수로 둔 상·중·하 비용표를 만든다. 공급자 가격 변동과 환율, 월 예산 상한, 호출 차단·축소 모드도 포함한다.

## 15. Redis 캐시를 기본 구성으로 둔 근거가 약하고 오답을 장시간 고정함

- **주장**: 학생 질문은 반복되므로 Redis 응답 캐시가 AI 비용과 응답 시간을 직접 줄이며 초기부터 채택해야 한다.
- **반론**: 반복률 측정이 없고, 동일 문자열 해시 캐시는 표현이 조금만 달라도 적중하지 않는다. 반대로 한 번 생성된 잘못된 날짜·담당 부서 답변은 24시간 반복 제공된다. 현재 키에는 문서 코퍼스 버전, 모델 버전, 검색 버전이 없다. Redis는 소규모 MVP에서 별도 장애·패치·메모리 운영 대상이다.
- **근거**: `infrastructure-options.md` 1장과 4.5는 반복 질문을 전제로 Redis를 채택한다. 그러나 현재 `CacheService.question_key()`는 정규화 질문 문자열의 SHA-256만 사용하고, 기본 TTL은 86,400초다. 문서가 권고한 버전 기반 키와 구현이 다르다.
- **위험도**: **높음**
- **수정 권고**: 초기에는 캐시 없이 실제 중복률과 비용을 측정하거나 프로세스 내 짧은 TTL 캐시로 시작한다. Redis를 유지한다면 코퍼스·검색·프롬프트·모델 버전을 키에 포함하고, 날짜·마감 질문은 TTL을 짧게 하며 공지 변경 시 관련 캐시를 무효화한다.

## 16. 질문 해시는 익명화가 아니며 캐시에도 개인정보가 남을 수 있음

- **주장**: 질문 원문과 전체 대화를 저장하지 않고, 해시 키와 TTL을 쓰면 개인정보 위험이 낮다.
- **반론**: 해시는 삭제·보존 정책을 대신하지 않는다. 학번·전화번호처럼 형식과 후보 공간이 좁은 값은 사전 대입이 가능하다. 더구나 캐시 값에는 생성 답변, 매치 공지, 분석된 필터가 통째로 저장되며 모델이 질문 속 개인정보를 답변에 되풀이할 수 있다. 세션 삭제 API는 세션 키만 지우고 질문별 캐시는 지우지 않는다.
- **근거**: `infrastructure-options.md` 5장은 질문 해시 키와 6~24시간 TTL을 권고한다. 현재 `ChatService`는 전체 응답 객체를 캐시하고, `CacheService.delete_session()`은 `session:{id}`만 삭제한다. 질문 사전 마스킹은 없다.
- **위험도**: **치명적**
- **수정 권고**: 입력 단계에서 학번·전화·이메일 등 패턴을 탐지해 외부 모델 전송과 캐시 저장 전에 차단·마스킹한다. 민감 가능 질문은 캐시하지 않는다. 캐시 값에서 답변 전문과 분석 필터를 최소화하고, 삭제 의미를 UI와 정책에 정확히 고지한다.

## 17. 상담 이관의 맥락 보존이 개인정보 과수집으로 바뀔 수 있음

- **주장**: 질문, 대화 요약, 인용 공지, 학생 동의 정보를 CRM/티켓에 넘기면 반복 설명을 줄일 수 있다.
- **반론**: 요약도 원문과 같은 개인정보일 수 있고, 생성 요약은 사실을 왜곡할 수 있다. 어느 부서가 어떤 내용을 볼 수 있는지, 민감 상담을 어떻게 분리할지, 보존기간과 삭제 권한이 무엇인지 정의되지 않았다. “사용자 동의” 하나로 과도한 전송이 정당화되지 않는다.
- **근거**: `global-university-chatbots.md` 4.4는 동의와 최소 정보 원칙을 언급하지만 필드별 전송 범위, 수신 권한, 보존기간, 정정 절차는 제시하지 않는다. 필수 보안 리뷰 문서도 없다.
- **위험도**: **치명적**
- **수정 권고**: 대화 전체 자동 전송을 금지하고, 사용자가 직접 편집·확인한 최소 필드만 보낸다. 민감 주제는 자동 분류·요약하지 않고 공식 보안 채널로 연결한다. 부서별 접근통제와 보존기간이 승인되기 전에는 티켓 기능을 출시하지 않는다.

## 18. 첨부파일 파이프라인은 MVP에 비해 과도하고 공격면이 큼

- **주장**: PDF, 스캔 이미지, HWP/HWPX, DOCX, XLSX/CSV, 선택적 ZIP까지 추출·색인하고 OCR 검수 큐를 둔다.
- **반론**: 형식별 파서·변환기·OCR·악성 파일 검사·샌드박스·페이지 인용·실패 재처리는 별도 문서 처리 제품 수준의 부담이다. 특히 HWP 변환기와 압축 파일은 취약점 및 자원 고갈 공격면을 늘린다. 현재 MVP의 핵심 질문 중 첨부 본문이 꼭 필요한 비율도 측정되지 않았다.
- **근거**: `rag-options.md` 6장과 13장 3단계는 광범위한 첨부 처리를 제안한다. `README.md`는 현재 통합 크롤러가 PDF/HWP/OCR 추출을 실행하지 않는다고 명시한다.
- **위험도**: **높음**
- **수정 권고**: MVP는 텍스트 PDF 한 종류, 크기 제한, 샌드박스 추출부터 시작한다. HWP·스캔·스프레드시트는 실제 실패 질문 비율이 임계치를 넘을 때 추가한다. 명단·성적·연락처 포함 문서는 기본 색인 제외한다.

## 19. PostgreSQL 작업 큐와 단일 워커도 ‘추가 서비스 없음’만큼 단순하지 않음

- **주장**: 저빈도 AI 작업에는 PostgreSQL 작업 테이블 + 단일 워커가 가장 단순하고 내구성 있다.
- **반론**: 직접 만든 큐는 lease 만료, 워커 사망 후 잠금 회수, 중복 실행, 우선순위 기아, 재시도 폭주, 스키마 마이그레이션, 관측 UI를 직접 책임져야 한다. 현재 `ProcessingJob`은 상태 기록일 뿐 실제 pending 소비·`SKIP LOCKED`·`available_at`·dead-letter 동작이 없다. 문서가 구현 난이도를 축소한다.
- **근거**: `infrastructure-options.md` 4.5는 필요한 큐 필드를 제시하지만 현재 `NoticeProcessor`는 요청 흐름에서 즉시 `running` 작업을 만들고 동기 처리한다. 별도 워커·스케줄러 컨테이너도 `docker-compose.yml`에 없다.
- **위험도**: **중간**
- **수정 권고**: MVP에서는 운영자가 실행하는 멱등 배치 명령과 실패 재실행으로 충분한지 먼저 검증한다. 자동 큐가 필요해지면 직접 구현과 검증된 큐 라이브러리의 총운영비를 비교하고, 최소한 lease 회수·중복 실행·dead 작업 테스트를 출시 조건으로 둔다.

## 20. Mac mini 직접 공개는 발표에는 쉬워도 실제 운영에는 부적합함

- **주장**: Mac mini Docker Compose는 제한된 MVP에 빠르고 저렴하며, Caddy·UPS·외부 백업을 보완하면 학교 서버로 이전할 수 있다.
- **반론**: 전교 대상 공개 서비스는 개인 장비의 전원·회선·물리 접근·OS 업데이트·원격 복구·방화벽·인증서 책임에 의존해서는 안 된다. 개발 장비와 운영 장비 분리, UPS, 외부 백업, 모니터링을 모두 추가하면 “저비용·단순” 이점도 줄어든다. 학교 정책상 외부 AI 송신이나 개인 장비 호스팅이 금지될 수도 있다.
- **근거**: `infrastructure-options.md` 4.3이 이 위험을 대부분 인정하면서도 최종 의사결정 표에서 초기 배포로 Mac mini Compose를 채택한다. 학교의 Docker 허용, 외부 송신, DNS/TLS 책임은 아직 미확인이다.
- **위험도**: **높음**
- **수정 권고**: Mac mini는 폐쇄된 데모·소수 테스터용으로만 명시하고 개인정보 입력을 금지한다. 공개 파일럿은 학교 승인 VM 또는 보안 심사를 통과한 관리형 환경에서만 수행한다. 이전성 주장은 실제 복원·전환 리허설로 검증한다.

## 21. 인프라 후보 점수는 객관적 평가처럼 보이지만 입력 근거가 없음

- **주장**: 경량·이전 가능형 A가 가중 합계 4.80으로 다른 후보보다 우월하다.
- **반론**: 가중치와 1~5점은 측정이나 이해관계자 합의 없이 작성자가 부여했다. 예를 들어 Redis와 직접 만든 DB 큐를 포함한 A에 운영 부담 5점을 준 것은 낙관적이며, 학교가 Spring 표준이면 B의 운영 부담과 이전성 점수는 크게 달라진다. 소수점 합계는 불확실성을 숨긴다.
- **근거**: `infrastructure-options.md` 2~3장은 가중치와 점수 산정의 실측 데이터·참여자를 제시하지 않는다. 학교 표준도 미확인이라고 적는다.
- **위험도**: **중간**
- **수정 권고**: 점수표를 임시 가설로 표시하고 범위값으로 바꾼다. 학교 전산 담당자, 개발자, 개인정보 담당자가 독립 채점한 뒤 비용 견적, 복구시간 시험, 운영 인력으로 보정한다.

## 22. 자료 유형의 고정 우선순위가 실제 효력 관계를 왜곡할 수 있음

- **주장**: 검색 자료 우선순위를 `학칙/규정 > 최신 공식 공지 > 부서 FAQ > 안내 페이지`로 두면 충돌을 안전하게 처리할 수 있다.
- **반론**: 상위 규정이 일반 원칙이고 최신 공지가 특정 학기 시행 예외·세부 일정·정정 내용을 담을 수 있다. 반대로 최신 게시일이 최신 효력을 뜻하지도 않는다. 문서 유형만으로 우선순위를 정하면 효력일, 개정 이력, 적용 대상, 정정 관계를 놓친다.
- **근거**: `korea-university-chatbots.md` 적용 제안 3번이 고정 순서를 제시한다. `rag-options.md`는 별도로 정정·대체 관계와 유효 기간이 필요하다고 인정한다.
- **위험도**: **높음**
- **수정 권고**: 문서 유형 우선순위를 제거하고 `authority`, `effective_from/to`, `supersedes`, `scope`, `approved_by`를 기준으로 충돌을 판정한다. 관계가 확인되지 않으면 둘 다 제시하고 담당 부서 확인을 요구한다.

## 23. “문장별 출처”가 정확성 보증처럼 오해될 수 있음

- **주장**: top chunk와 문장별 출처·원문 구간을 제시하면 근거 품질이 매우 높아진다.
- **반론**: 인용 링크는 해당 문장이 출처에 실제로 함의되는지 보장하지 않는다. 여러 공지의 조건을 합성하거나 대상·예외를 생략한 답에도 그럴듯한 인용을 붙일 수 있다. 자동 문장별 정렬은 추가 모델 호출 또는 복잡한 검증을 요구한다.
- **근거**: `rag-options.md`는 E의 근거 제시를 `매우 높음`으로 평가하고 문장별 출처를 권장하지만, citation entailment 평가나 합성 답변의 조건 보존 검사는 구체화하지 않는다.
- **위험도**: **높음**
- **수정 권고**: MVP는 생성 요약보다 근거 구간을 짧게 그대로 보여 주고, 날짜·금액·대상·서류 같은 원자적 필드는 추출값과 원문을 함께 검증한다. citation precision뿐 아니라 인용이 답을 실제 지지하는지 담당자 평가를 포함한다.

## 24. KPI 목록이 운영 역량과 데이터 접근 권한을 초과함

- **주장**: 답변 품질부터 신청 완료율, 문의시간 절감, 형평성, 신뢰까지 폭넓은 KPI를 단계별로 관리해야 한다.
- **반론**: 신청·납부 완료율과 집단별 해결률은 학사 데이터 결합, 기준선, 동의·법적 근거, 실험 설계가 필요하다. 초기 MVP가 이 모든 지표를 수집하면 개인정보 범위와 분석 부담만 커질 수 있다. 지표가 많으면 실제 출시 판단 기준도 불명확해진다.
- **근거**: `global-university-chatbots.md` 4.7은 20개에 가까운 KPI를 제안하지만 수집 가능성, 데이터 소유자, 최소 표본, 성공·중단 기준을 제시하지 않는다.
- **위험도**: **중간**
- **수정 권고**: MVP 게이트를 `정답 공지 Recall@5`, `날짜·대상 조건 위반율`, `근거 없는 확정 답변율`, `과업 성공률`, `재사용 의향`의 5개 이내로 줄인다. 개인 학사 데이터가 필요한 KPI는 별도 승인 후 익명 집계로만 추가한다.

## 우선 수정 순서

1. 누락된 보안 리뷰와 UX 권고안을 복구·작성하고 공개 범위를 다시 승인한다.
2. 최근 2개월 제한을 해소할 백필·유효기간 정책과 학교 학사기간 테이블을 만든다.
3. 날짜·학기·대상·만료 상태의 결정적 테스트와 100~300개 검색 평가셋을 먼저 구축한다.
4. 캐시의 개인정보·오답 고정 문제를 해결하고, 필요성이 입증될 때만 Redis를 기본 구성으로 둔다.
5. C/D/E 검색 방식을 동일 데이터로 비교한 뒤 목표 아키텍처를 결정한다.
6. MVP에서는 티켓, SSO, CRM, 선제 알림, 업무 실행, 광범위 첨부 처리를 제외한다.
7. 학교 승인 인프라와 부서별 지식 책임자가 확보된 뒤 제한된 공개 파일럿을 진행한다.

## 출시 차단 조건

다음 항목 중 하나라도 충족되지 않으면 “공식 학사 답변 서비스”로 공개해서는 안 된다.

- 보안 리뷰와 개인정보 처리 흐름이 승인되지 않음
- 학사기간 경계, 계절학기, 연도 생략, 정정 공지 테스트가 통과하지 않음
- 최근 2개월 밖의 현행 규정·상시 공지를 검색할 수 없음
- 검수되지 않은 메타데이터 hard filter가 정답을 제거할 수 있음
- 질문·답변 캐시의 민감정보 차단 및 삭제 정책이 없음
- 실제 부서 연락처·지식 책임자·오답 정정 절차가 없음
- 학생 과업 테스트에서 기존 공지 검색보다 명확한 이점이 확인되지 않음
