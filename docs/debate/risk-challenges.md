# 라운드 4 — 보안 에이전트와 비판적 리뷰어의 위험 도전

- 작성일: 2026-07-20
- 참여 역할: Security Architect, Critical Reviewer
- 공격 대상: `React + FastAPI + PostgreSQL + pgvector + 조건부 OpenAI` MVP안

## 1. 판정

후보 A는 비교 후보 중 가장 단순하지만 아직 공식 학사 답변 서비스로 출시할 상태는 아니다. 단일 PostgreSQL이 공격면과 데이터 사본을 줄이는 것은 장점이지만, 검색 품질·데이터 최신성·관리 경계·개인정보·비용 통제가 자동으로 해결되지는 않는다.

따라서 후보 A를 **검증용 제한 파일럿의 기준 구성**으로 채택하고, 이 문서의 출시 게이트를 통과한 뒤에만 공식 MVP로 승격한다.

## 2. 위협 모델

### 보호 자산

- 공식 공지의 원문, 버전, 유효 상태와 정정·삭제 관계
- 답변의 정확성, 근거, 최신성과 사용자 신뢰
- 질문에 포함될 수 있는 학번, 연락처, 이메일, 건강·재정·징계 정보
- OpenAI 키, DB 자격 증명, 관리자 세션, 백업
- AI 호출 예산, DB·워커 처리 용량, 서비스 가용성
- 프롬프트·모델·임베딩·청커 버전과 평가·감사 이력

### 신뢰 경계

```text
익명 사용자 → 공개 프록시/API → PostgreSQL → 외부 AI

관리자/스케줄러 → 인증된 관리 경계 → 크롤러
  → 대학 웹·첨부(비신뢰) → 제한 파서 → 검수 대기 색인
  → 검증 후 활성 인덱스
```

공개 공지는 비밀은 아니지만 높은 무결성이 필요한 비신뢰 입력이다. 공지 본문과 첨부 안의 명령을 시스템 지시로 처리해서는 안 된다.

## 4. 필수 10개 쟁점에 대한 공격과 방어 조건

### PostgreSQL + pgvector만으로 충분한가?

처리 규모에는 충분할 가능성이 높지만 한국어 정확도를 보장하지 않는다. `ILIKE/trigram`, PostgreSQL FTS, vector, 조합 검색을 실제 학생 표현·오타·띄어쓰기·학수번호·서식명이 포함된 같은 평가셋으로 비교해야 한다. Recall@5, 조건 위반율, p95를 만족하지 못하면 “충분” 판정을 취소한다.

### Elasticsearch가 MVP에 필요한가?

초기에는 불필요하다. 다만 한국어 평가 목표 미달을 무시한 채 PostgreSQL만 고집해서는 안 된다. PostgreSQL 실패 질의 중 Nori/BM25가 회수하는 비율과 운영 비용을 비교하는 재평가 트리거를 둔다.

### Redis는 처음부터 필요한가?

필요하지 않다. 현재 질문 문자열 해시와 24시간 응답 캐시는 공지 변경 후 오답과 개인정보를 고정한다. Redis를 넣으려면 실제 적중률, 비용 절감, 세대/모델/프롬프트/권한 버전 키, 변경 무효화와 민감질의 제외가 먼저다.

### 공지를 전부 chunking해야 하는가?

전면 청킹과 공지 단일 임베딩 모두 극단적이다. 길이·구조별 정책을 비교하고 근거 Recall과 인용 정확도가 좋아지는 문서에만 청킹한다. parent-child 묶음, 인접 청크 병합, 삭제/ACL 상속이 필수다.

### metadata와 vector 중 무엇이 우선인가?

권한·공개 범위·검증된 효력일은 metadata hard filter가 맞다. 그러나 미검수 LLM 추출값은 공격 가능하고 오분류될 수 있다. 필드별 confidence, provenance, 다중값, hard-filter false-negative 회귀 테스트 없이는 후보 제거에 쓰지 않는다.

### 질문 분석에 LLM이 항상 필요한가?

아니다. 매번 호출하면 PII 반출·비용·지연·비결정성이 늘어난다. 규칙·학사기간 테이블·용어 사전을 기본으로 하고, 복수 해석이 남을 때만 LLM을 사용한다. OpenAI 장애에도 원문 검색과 FAQ가 동작해야 한다.

### FAQ와 RAG의 우선순위는?

검수 FAQ가 우선이지만 소유자·유효기간·원문 버전·갱신 SLA가 없으면 FAQ 자동답변을 끈다. RAG 근거와 충돌하면 합성하지 않고 FAQ를 비활성화하거나 두 근거와 담당 부서를 제시한다.

### 공지 수정과 삭제를 어떻게 감지할 것인가?

본문 해시만으로는 불충분하다. 전체 수집 완료성, 페이지/건수 급감, 첨부 교체, DOM 변경, 일시 누락, 복원 시나리오를 테스트한다. 대량 변화는 자동 활성화하지 않고 이전 인덱스 세대를 유지한다.

### 검색 결과가 없다는 판단은?

고정 cosine 값 하나로 정하지 않는다. 후보 부재, 명시 조건 불일치, 필수 claim을 지지하는 span 부재, 충돌, 권위·효력 불확실, stale-only를 분리한다. 골든셋에는 답이 실제로 없는 질문과 범위 밖 질문을 포함해 abstention precision/recall을 측정한다.

### 생성 없이 답변 가능한 질문은?

검수 FAQ, 담당 부서와 연락처, 공지 제목·게시일·원문 링크, 검증된 신청 기간, 정확 문서/학수번호 조회, 서비스 범위와 장애 안내는 템플릿으로 처리한다. 생성 모델이 없으면 이 기능도 중단되는 구조는 출시 차단 사유다.

## 5. 출시 차단 조건

다음 중 하나라도 미충족이면 공식 서비스로 공개하지 않는다.

1. 관리 API 인증·인가·감사와 동시 실행 제한이 없음
2. 외부 모델 호출 전 PII 차단/마스킹이 없음
3. 프롬프트 인젝션·악성 공지·무근거 답변 회귀 테스트가 없음
4. 부분 수집·건수 급락이 대량 삭제로 자동 반영될 수 있음
5. 검색·무응답 임계값이 100~300개 골든셋으로 검증되지 않음
6. AI rate limit, timeout, 예산 상한, kill switch가 없음
7. 기본 자격 증명, 가변 이미지, 비밀 회전 부재
8. 외부 AI 처리·보존·국외 이전에 대한 학교 승인이 없음
9. 백업 복원, 인덱스 세대 롤백, AI 장애 fallback 시험이 없음
10. 첨부 처리에 allowlist, 샌드박스, 크기/시간/PII 제한이 없음
11. 현행 규정·정정 공지·학사기간 백필이 없음
12. 지식 소유자, 오답 정정 SLA, 사고 대응 책임자가 없음

## 6. 비판적 결론

후보 A의 높은 점수는 “검증된 정확도”가 아니라 현재 코드 재사용과 적은 운영 대상에서 온다. 이 이점을 이유로 품질·보안을 낙관해서는 안 된다. 후보 A는 최종 선택이되, 위 검증 게이트와 재평가 트리거가 아키텍처의 일부여야 한다.
