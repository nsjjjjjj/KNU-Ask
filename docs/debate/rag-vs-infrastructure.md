## 1. 각 에이전트의 첫 제안

### RAG 에이전트

RAG 에이전트는 메타데이터 필터와 의미 검색을 결합하고, 검증 결과에 따라 PostgreSQL 안에서 lexical 검색까지 추가하는 경로를 제안했다. 날짜·학기·학년·유효기간은 벡터 유사도에 맡기지 않고 구조화 필드로 처리하며, 긴 문서만 구조를 보존해 청킹한다. 생성 모델은 충분한 근거가 있을 때의 요약 단계로 제한한다.

### 인프라 에이전트

인프라 에이전트는 기존 Python 코드 재사용, 단일 기준 데이터, 적은 운영 대상을 이유로 FastAPI와 PostgreSQL/pgvector를 제안했다. 최초 안에는 Redis 응답 캐시와 PostgreSQL 작업 큐가 포함됐으나, 반복률과 작업량이 측정되지 않았다는 비판을 수용해 둘 다 MVP 필수 구성에서 제외했다.

## 2. 서로의 제안에 대한 검토

### RAG가 인프라안에 제기한 쟁점

1. Redis는 적중률이 검증되지 않았고 오래된 날짜 답변과 개인정보를 고정할 수 있다.
2. 직접 만든 PostgreSQL 큐도 lease, 중복 실행, 재시도, dead job을 운영해야 하므로 단순하다고 단정할 수 없다.
3. PostgreSQL이 FTS를 지원한다는 사실과 한국어 검색 품질이 충분하다는 주장은 다르다.
4. 인프라 후보 점수는 학교 표준·인력·복구 시험이 반영되기 전까지 가설이다.

### 인프라가 RAG안에 제기한 쟁점

1. 처음부터 모든 공지를 청킹하고 하이브리드 검색을 구축하면 평가 전에 복잡도가 커진다.
2. 미검수 LLM 메타데이터를 hard filter로 사용하면 정답을 제거할 수 있다.
3. Elasticsearch나 별도 Vector DB는 원본 DB와 삭제·수정 동기화를 새로 책임져야 한다.
4. 검색 LLM, reranker, 답변 LLM을 항상 호출하는 구조는 지연·비용·장애 지점을 불필요하게 늘린다.

## 3. 필수 쟁점별 토론과 결론

| 쟁점 | RAG 입장 | 인프라 입장 | 합의된 MVP 결정 |
|---|---|---|---|
| PostgreSQL + pgvector만으로 충분한가 | 수천 건~초기 수만 청크에는 충분하나 한국어 품질은 평가 필요 | 단일 백업·트랜잭션·삭제 정합성이 유리 | **충분하다고 가정하고 시작하되 골든셋으로 검증** |
| Elasticsearch가 MVP에 필요한가 | 정확어 검색 이점은 있으나 PG lexical baseline을 먼저 검증 | JVM·샤드·스냅샷·이중 색인이 과도 | **도입하지 않음** |
| Redis는 처음부터 필요한가 | 오래된 오답과 민감정보 캐시 위험 | 반복률·절감 효과가 미측정 | **도입하지 않음** |
| 공지를 전부 chunking해야 하는가 | 긴·복합 공지만 구조 보존 청킹 | 짧은 공지 청킹은 비용과 운영량만 증가 | **선택적 청킹** |
| metadata와 vector 중 무엇이 우선인가 | 명시 조건과 유효성은 metadata, 주제는 lexical/vector | 검수·신뢰도가 낮으면 hard filter 금지 | **신뢰도 높은 metadata 우선, 나머지는 boost** |
| 질문 분석에 LLM이 항상 필요한가 | 규칙 실패·복합 의도에서만 필요 | 매번 호출하면 비용·외부 전송·지연 증가 | **규칙/사전 우선, 조건부 LLM** |
| FAQ와 RAG의 우선순위 | 검수된 고신뢰 FAQ 우선, 그 외 RAG | 대규모 FAQ 운영 시스템은 시기상조 | **소수 검수 FAQ → 구조화 답변 → RAG** |
| 수정·삭제 감지 | 안정 ID, 본문/첨부 해시, 버전 보존 | 목록 전체 대조와 반복 누락 확인 필요 | **해시 변경 + 목록 누락 재확인 + 이력 보존** |
| 검색 결과 없음 판단 | 단일 cosine 임계값 금지 | 조건 충족과 근거 존재를 별도 판정 | **다중 신호와 상태 코드 사용** |
| 생성 없이 답할 질문 | 날짜·상태·링크·부서·목록·검수 FAQ | 템플릿 응답이 더 빠르고 재현 가능 | **결정적 응답 경로를 P0로 구현** |

## 4. 상세 합의

### 4.1 PostgreSQL + pgvector의 범위

PostgreSQL은 공지 원문, 문서 버전, 구조화 메타데이터, lexical 필드, 벡터, 처리 상태의 기준 데이터다. MVP에서는 exact vector search로 기준선을 만들고, 데이터와 p95 지연이 필요성을 보일 때 HNSW를 추가한다. 별도 검색 계층 재평가 조건은 다음과 같다.

- PostgreSQL 조합 검색이 검수 질문의 Recall@5 목표를 지속적으로 충족하지 못함
- 형태소 분석·동의어·오탈자 처리가 주요 실패 원인으로 확인됨
- 코퍼스나 트래픽 증가로 검색 지연 SLO를 반복 위반함
- 별도 검색 계층을 운영할 인력과 백업·재색인 절차가 확보됨

### 4.2 PostgreSQL lexical 검색의 위치

PostgreSQL 기본 FTS의 한국어 성능을 미리 높게 평가하지 않는다. 제목/정확 구문, `ILIKE`, `pg_trgm`, FTS, pgvector, 결합 검색을 동일한 100~300개 골든 질문으로 비교한다. MVP 아키텍처는 PostgreSQL 한 개를 선택하지만, 그 안의 최종 랭킹 공식은 평가 결과로 결정한다.

### 4.3 선택적 청킹

- 짧은 HTML 공지: 공지 전체를 하나의 검색·근거 단위로 유지
- 긴·복합 공지: 제목, 소제목, 목록, 표 경계를 보존해 약 350~800 token으로 분할
- 표: 헤더와 공통 학기·대상 문맥을 각 청크에 보존
- 텍스트 PDF: 크기·형식 제한과 추출 신뢰도 기록 후 처리
- HWP, 스캔, ZIP, 명단·성적 파일: MVP 자동 색인에서 제외
- 검색 결과: 청크를 그대로 여러 장 보여주지 않고 `notice_id`로 묶어 공지 카드와 근거 구간을 반환

### 4.4 metadata hard/soft 정책

항상 hard filter인 항목은 공개 범위와 접근 권한이다. 학년도·학기·학년·대상은 사용자가 명시했고 해당 문서 필드가 검수됐거나 신뢰도가 충분할 때만 hard filter로 쓴다. 추론된 학기, 미검수 LLM 필드, 누락 가능성이 높은 카테고리는 soft boost로 사용한다. hard filter가 0건이면 사용자가 명시하지 않은 조건만 완화하며, 명시 조건을 몰래 제거하지 않는다.

### 4.5 질문 분석과 답변 모드

규칙·사전은 날짜, 학년도, 학기, 학년, 공지 번호, 부서명, 학수번호, URL/연락처 요청을 먼저 처리한다. LLM은 복합 조건, 생략된 대명사, 여러 의도처럼 규칙 결과가 불충분한 경우에만 호출한다. 답변 모드는 다음으로 분리한다.

- `faq`: 승인·유효한 FAQ 그대로 반환
- `deterministic`: 구조화 필드와 템플릿으로 반환
- `generated`: 검증된 근거를 짧게 요약
- `search_results_only`: 생성 없이 관련 공지와 근거 제시
- `department_handoff`: 공개 자료로 판단할 수 없어 담당 부서 안내

### 4.6 FAQ 우선순위와 무효화

FAQ는 자동 생성해 바로 게시하지 않는다. 각 FAQ는 `source_document_id`, `source_version`, `source_hash`, `effective_from/to`, `review_status`, `reviewed_by/at`를 가진다. 원문 해시가 변하거나 유효기간이 끝나면 즉시 비활성화한다. 정확·고신뢰 일치가 아니면 공지 검색으로 내려간다.

### 4.7 공지 변경과 삭제 감지

1. 게시판별 `source_id + external_notice_id` 또는 canonical URL을 안정 식별자로 저장한다.
2. `ETag`, `Last-Modified`, 원문 수정시각은 최적화 힌트로만 사용한다.
3. 정규화 본문과 첨부 manifest/파일 각각 SHA-256을 계산한다.
4. 해시가 바뀌면 기존 레코드를 덮지 않고 새 `document_version`을 만든다.
5. 변경된 청크만 재구조화·재임베딩하고 검증 후 새 인덱스 세대를 활성화한다.
6. 전체 목록 수집이 성공했을 때만 누락을 기록한다.
7. 404/410 확인 또는 연속 2~3회 성공한 전체 대조에서 누락될 때 `archived/deleted`로 전환한다.
8. 정정·대체 관계는 `corrects/supersedes`로 연결하고 이전 버전을 보존한다.
9. 연결된 FAQ와 향후 캐시를 무효화한다.

### 4.8 검색 실패 판정

검색 실패는 결과 개수나 cosine 점수 하나로 정하지 않는다. 명시 조건 충족, 질문 유형별 필수 근거 구간, 최신성, 문서 권위와 정정 관계, 추출 신뢰도, 상위 결과 간 충돌을 함께 본다.

| 상태 | 판단 |
|---|---|
| `no_result` | 관련 후보 자체가 없음 |
| `constraint_mismatch` | 관련 후보는 있으나 명시 조건 충족 후보가 없음 |
| `insufficient_evidence` | 후보는 있으나 핵심 날짜·자격·절차 근거가 없음 |
| `conflicting_evidence` | 유효한 공식 문서들이 서로 충돌 |
| `stale_only` | 종료·보관 자료만 존재 |
| `out_of_scope` | 현재 수집·서비스 범위 밖 |
| `clarification_required` | 조건 하나가 빠져 답이 달라짐 |
| `system_error` | 검색 또는 외부 서비스 장애 |

임계값은 골든셋에서 무응답 정밀도, Recall@5, 날짜·대상 조건 위반율, 근거 없는 확정 답변율로 보정한다. 평가 전에는 오답보다 무응답을 선호한다.

## 5. 최종 합의와 남은 검증

두 에이전트는 **후보 A: React + FastAPI + PostgreSQL + pgvector + OpenAI**를 MVP 기준으로 합의했다. 단, OpenAI는 모든 질문에 필수인 계층이 아니며 Redis, Elasticsearch, 별도 Vector DB는 제외한다.

출시 전에 반드시 검증할 항목은 다음과 같다.

1. 최근 2개월이 아니라 최소 1~2개 학년도와 현행 규정 백필
2. 강남대학교 학사기간 테이블과 KST 경계 테스트
3. 필드별 추출 신뢰도와 다중값 metadata 스키마
4. 한국어 검수 질문 100~300개로 검색 방식 비교
5. 공지 수정·삭제·정정 후 검색과 FAQ 갱신 SLA
6. 비생성 경로와 OpenAI 장애 fallback
