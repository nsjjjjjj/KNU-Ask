# 대학 공지 RAG 및 검색 아키텍처 비교

## 1. 결론

대학 공지 검색의 기본 아키텍처로 **E. 메타데이터 필터 + PostgreSQL 전문 검색 + pgvector**를 권장한다. 다만 MVP 첫 단계에서는 현재 코드와 가장 가까운 **D. 메타데이터 필터 + 벡터 검색**으로 시작하고, 검색 품질 평가 세트가 준비되는 즉시 E로 확장하는 편이 안전하다. 반복 질문이 충분히 쌓인 뒤에는 E 앞에 **G. FAQ 우선 검색**을 선택적 캐시 계층으로 추가한다.

핵심 판단은 다음과 같다.

- 날짜, 학년도, 학기, 학년, 재학 상태, 공지 만료 여부는 임베딩 유사도에 맡기지 않고 **구조화된 메타데이터로 먼저 제한하거나 점수화**한다.
- 검색의 기본 반환 단위는 공지 전체가 아니라 **공지(parent)와 근거 구간(chunk)의 조합**이다. 최종 답변에는 선택된 구간과 제목·기간·대상·원문 URL을 함께 전달한다.
- 제목, 정확한 행정 용어, 학수번호, 서식명에는 전문 검색이 강하고, 학생의 구어체·동의어에는 벡터 검색이 강하다. 두 결과를 RRF(Reciprocal Rank Fusion)나 학습된 재정렬로 결합할 가치가 충분하다.
- 종료 공지는 기본적으로 삭제하지 않는다. 현재 신청 질문에서는 낮은 점수를 주되, 과거 이력 질문이나 특정 연도 질문에서는 다시 정상 후보가 되도록 **질의 의도에 따라 감점**한다.
- 검색 실패는 단일 유사도 임계값만으로 판단하지 않는다. 후보 점수, 상위 결과 간 격차, 필수 조건 충족, 근거 문장 존재, 최신성, 추출 신뢰도를 함께 사용한다.

권장 진화 경로는 다음과 같다.

```text
MVP: D (metadata + vector)
  -> 운영 기본: E (metadata + full-text + vector)
  -> 반복 문의가 많을 때: G (FAQ gate + E)
  -> 관계형 질문이 핵심 KPI가 될 때만: F를 보조 인덱스로 추가
```

## 2. 전제와 평가 범위

이 문서는 다음과 같은 대학 공지 질의를 대상으로 한다.

- “2026학년도 2학기 3학년 수강신청 기간이 언제야?”
- “복학생도 국가장학금 신청할 수 있어?”
- “등록금 분할납부 서류가 뭐야?”
- “작년 교직 이수 신청 공지 보여줘.”
- “마감 지난 예비군 공지 말고 지금 신청 가능한 것만 알려줘.”

현재 저장소는 PostgreSQL 16, pgvector, FastAPI를 사용하며 공지별 구조화 메타데이터와 단일 임베딩을 저장한다. 현재 검색기는 보관 공지를 제외하고 일부 메타데이터를 SQL로 필터한 뒤 벡터, 단순 키워드 일치, 상태 점수를 애플리케이션에서 합산한다. 이 문서의 권고안은 기존 구조를 폐기하기보다 다음 기능을 보완하는 방향이다.

- PostgreSQL `tsvector`/GIN 기반 전문 검색
- 공지 단위와 chunk 단위의 분리
- 학년·대상·기간 필터의 실제 적용
- 현재 시각에 따른 상태 재계산
- 검색 실패 및 근거 충분성 판정
- 첨부파일 추출·색인 파이프라인
- 평가 데이터에 기반한 임계값 보정

비용 표기는 특정 공급자의 변동 가격 대신 상대 등급과 비용 발생식을 사용한다.

## 3. A~G 전체 비교

등급은 이 서비스의 공지량과 질의 특성을 기준으로 한 상대 평가다. 정확도·처리 능력·변경 대응·근거 제시는 높을수록 좋고, 비용·속도·난이도는 낮을수록 좋다.

| 후보 | 정확도 | 날짜 처리 | 학년도·학기 | 학년 필터 | 만료 처리 | 토큰 비용 | 응답 속도 | 구현 난이도 | 운영 난이도 | 변경 대응 | 근거 제시 |
|---|---|---|---|---|---|---|---|---|---|---|---|
| A. 전체 원문을 매번 LLM 전달 | 소량일 때 중, 규모 증가 시 하 | LLM 의존, 불안정 | LLM 의존 | LLM 의존 | LLM 의존 | 매우 높음 | 느림 | 낮음~중간 | 높음 | 재수집만 하면 됨 | 중, 인용 위치 불명확 |
| B. 벡터 검색만 RAG | 중 | 낮음 | 낮음~중 | 낮음 | 낮음 | 낮음 | 빠름 | 낮음 | 낮음~중 | 임베딩 갱신 필요 | 중 |
| C. 키워드 검색만 | 중 | 규칙 결합 시 높음 | 규칙 결합 시 높음 | 규칙 결합 시 높음 | 규칙 결합 시 높음 | 매우 낮음 | 매우 빠름 | 낮음 | 낮음 | 즉시 반영 | 높음 |
| D. 메타데이터 + 벡터 | 높음 | 높음 | 높음 | 높음 | 높음 | 낮음 | 빠름 | 중 | 중 | 메타데이터·임베딩 갱신 | 높음 |
| E. 메타데이터 + 전문 + 벡터 | 매우 높음 | 높음 | 높음 | 높음 | 높음 | 낮음 | 빠름~중간 | 중~높음 | 중 | 각 인덱스 증분 갱신 | 매우 높음 |
| F. 지식 그래프 | 관계 질의는 매우 높음, 일반 검색은 중 | 높음 | 높음 | 높음 | 높음 | 중 | 중간 | 매우 높음 | 매우 높음 | 관계 재추출 필요 | 매우 높음 |
| G. FAQ 우선 + 공지 RAG | 반복 질문은 매우 높음, 신규 질문은 하위 RAG와 동일 | FAQ 최신성에 좌우 | FAQ 스키마에 좌우 | FAQ 스키마에 좌우 | 동기화 시 높음 | 매우 낮음~낮음 | 매우 빠름~빠름 | 중~높음 | 높음 | FAQ 무효화 필요 | 높음 |

### A. 전체 공지 원문을 매번 LLM에 전달

작은 데모에서는 구현이 단순하지만, 공지가 늘면 컨텍스트 한도, 비용, 지연 시간이 급격히 악화된다. 관련 없는 공지가 많아질수록 LLM이 정확한 날짜와 대상을 혼동할 가능성도 커진다. 공지 수가 수십 건 이하인 임시 검증을 제외하면 채택하지 않는다.

### B. 벡터 검색만 사용하는 RAG

“등록금 나눠 내기”와 “분할납부” 같은 의미 대응에는 유리하다. 그러나 `2025`, `2학기`, `3학년`, “마감 전” 같은 조건은 의미 유사도만으로 보장되지 않는다. 날짜와 대상이 답의 정확성을 좌우하는 대학 공지에는 단독 사용이 부적합하다.

### C. 키워드 검색만 사용하는 방식

정확한 공지명, 부서명, 학수번호, 서식명에는 강하고 저렴하다. 반면 학생 표현과 행정 용어가 다르거나 띄어쓰기·복합어·동의어가 다양한 한국어 질의에서 재현율이 떨어진다. 관리 화면의 정확어 검색이나 벡터 장애 시 fallback으로는 유용하다.

### D. 메타데이터 필터 + 벡터 검색

시간·대상 조건을 구조적으로 지키면서 구어체 검색을 지원한다. 현재 저장소와 가장 가까워 MVP에 적합하다. 다만 고유명사와 정확 구문을 벡터가 놓칠 수 있고, 필터를 너무 엄격하게 적용하면 메타데이터 추출 오류 때문에 정답을 제거할 수 있다.

### E. 메타데이터 필터 + 전문 검색 + 벡터 검색

정확 용어와 의미 검색을 동시에 사용한다. PostgreSQL 전문 검색은 `tsvector`와 `tsquery`의 `@@` 연산으로 일치 여부를 판단하고, `ts_rank_cd`로 위치 정보를 고려한 순위를 계산할 수 있다. pgvector도 PostgreSQL 전문 검색과 결합하고 RRF 또는 cross-encoder로 결과를 합치는 방식을 공식적으로 안내한다. 이 서비스의 운영 기본안이다.

단, PostgreSQL 기본 설정만으로 한국어 형태소 분석이 완벽해지는 것은 아니다. 초기에는 제목·본문의 정규화된 토큰, 공백 제거 보조 필드, 동의어 사전을 활용하고, 실제 평가에서 필요성이 확인되면 PGroonga 또는 별도 한국어 분석기를 검토한다.

### F. 지식 그래프 기반 검색

`공지 -> 제도 -> 대상 -> 학기 -> 담당 부서 -> 필요 서류` 관계를 명시하면 “복학생이 신청 가능한 장학 제도 중 2학기 마감 전인 것” 같은 다단계 관계 질의에 강하다. 그러나 공지 검색 대부분은 문서 검색과 필터로 해결되며, 그래프 스키마·엔터티 정규화·관계 변경 관리 비용이 크다. 단독 검색기가 아니라 E의 보조 후보 생성기 또는 설명 계층으로만 장기 검토한다.

### G. FAQ 우선 검색 후 공지 RAG

반복 질문의 비용과 지연을 크게 줄인다. 그러나 FAQ 답변이 원본 공지 변경 후 남아 있으면 오래된 답을 매우 빠르게 제공하는 위험이 있다. FAQ는 반드시 `source_notice_id`, 원문 해시, 유효 기간, 검수 상태를 가져야 하며, 공지가 변경되거나 만료되면 자동 무효화해야 한다. FAQ가 높은 신뢰도로 일치하지 않으면 E로 즉시 내려간다.

## 4. 상세 아키텍처 후보

### 후보 1: D — 메타데이터 필터 + 공지 단위 벡터 검색

#### 데이터 흐름

```text
공지 HTML/API 수집
  -> 본문 정규화 + 해시 비교
  -> 규칙 추출(날짜/학년도/학기/학년)
  -> LLM 구조화(분류/대상/행동/동의어, 규칙 결과 검증)
  -> 공지별 search_text 생성
  -> 공지별 임베딩 생성 및 pgvector 저장

사용자 질문
  -> 질의 규칙 분석 + 필요 시 LLM 보완
  -> metadata hard/soft filter
  -> pgvector top 30
  -> 상태·최신성·조건 일치 재정렬
  -> top 3~5 원문/근거를 LLM에 전달
  -> 답변 + 공지 제목 + URL + 기간 제시
```

#### 구성요소

- 수집기와 변경 감지기
- 규칙 기반 시간·대상 파서
- 구조화 LLM과 스키마 검증기
- PostgreSQL 메타데이터 테이블
- pgvector 공지 임베딩 테이블
- 필터·재정렬 검색 서비스
- 답변 생성기와 출처 렌더러
- 검색 품질 로그

#### 장점

- 현재 코드 구조를 대부분 재사용할 수 있다.
- 날짜, 학기, 학년, 상태 조건을 벡터 유사도와 분리할 수 있다.
- 질문마다 소수 공지만 LLM에 전달하므로 비용이 낮다.
- 공지 원문 URL과 메타데이터를 함께 제시하기 쉽다.

#### 단점

- 한 공지에 여러 주제나 표가 포함되면 단일 임베딩이 세부 내용을 희석한다.
- 정확한 서식명, 코드, 복합 명칭 검색을 놓칠 수 있다.
- 잘못 추출된 필드를 hard filter로 쓰면 정답이 후보에서 사라진다.
- 현재 코드처럼 후보를 먼저 30개로 자른 뒤 조건 점수를 더하면, 조건에 맞는 문서가 벡터 top 30 밖에 있을 때 회수하지 못한다.

#### 예상 비용

- 색인: 공지 신규·변경 시 `구조화 LLM 1회 + 임베딩 1회`.
- 질의: `질의 분석 0~1회 + 질문 임베딩 1회 + 답변 LLM 1회`.
- 저장: 공지 수에 선형. 1,536차원 float32 벡터 원자료는 공지당 약 6KB이며 인덱스·행 오버헤드는 별도다.
- 상대 등급: 구축비 **중**, 질의비 **낮음**, 운영비 **중**.

#### 구현 난이도

중간. 현재 구현에 학년·대상 조건, 시간 상태 재계산, 필터 완화 재시도, 평가 로그를 추가하면 된다.

#### MVP 적합성

**매우 높음.** 데이터가 수천 건 이하이고 검색 품질을 빠르게 검증하는 단계에 적합하다.

#### 확장 적합성

**중간.** chunk 검색과 전문 검색이 필요해질 때 스키마 확장이 필요하지만, E로 자연스럽게 이동할 수 있다.

### 후보 2: E — 메타데이터 + PostgreSQL 전문 검색 + chunk 벡터 검색

#### 데이터 흐름

```text
공지/첨부 수집
  -> 파일별 해시·버전 관리
  -> 텍스트/표 추출 및 정규화
  -> 규칙 추출 -> LLM 보완 -> 검증/검수 큐
  -> notice(parent) 저장
  -> 의미 단락별 chunk(child) 생성
  -> 제목·본문·메타데이터 tsvector/GIN 생성
  -> chunk 임베딩/HNSW 또는 exact vector 저장

사용자 질문
  -> 기준 시각·학년도·학기·대상 정규화
  -> hard filter(명시 조건) + soft boost(추정 조건)
  -> 병렬 후보 생성
       1) PostgreSQL 전문 검색 top N
       2) pgvector 의미 검색 top N
  -> RRF 결합
  -> 메타데이터·만료·최신성 재정렬
  -> 같은 공지의 chunk 병합 및 중복 제거
  -> 근거 충분성/검색 실패 판정
  -> top 3 공지의 top chunk를 답변 LLM에 전달
  -> 문장별 출처·원문 URL 제공
```

#### 구성요소

- 후보 1의 모든 구성요소
- `notice_chunks` 테이블: `notice_id`, `section_path`, `chunk_text`, `token_count`, `embedding`, `content_hash`
- `search_document tsvector` 생성 열 또는 트리거와 GIN 인덱스
- 공지/필드별 가중치: 제목 A, 키워드·서식명 B, 본문 C, 첨부 추출문 D
- 전문 검색과 벡터 검색의 RRF 결합기
- parent-child 병합기와 선택적 cross-encoder/LLM reranker
- 검색 실패 판정기 및 오프라인 평가 도구

#### 장점

- 정확한 행정 용어와 자연어 표현을 모두 회수한다.
- 긴 공지에서도 답과 가까운 근거 구간을 찾는다.
- 전문 검색 결과는 일치 단어와 구간을 보여주기 쉬워 근거 제시가 좋다.
- PostgreSQL 하나로 필터, 전문 검색, 벡터 검색, 트랜잭션을 운영할 수 있어 MVP 이후에도 인프라가 과도하게 늘지 않는다.
- 공지 변경 시 해당 공지와 변경 chunk만 다시 색인할 수 있다.

#### 단점

- 두 검색 점수의 스케일이 달라 단순 가중합은 튜닝이 어렵다.
- RRF의 `k`, 후보 수, 필터 강도, 상태 감점을 평가 데이터로 조정해야 한다.
- 한국어 전문 검색 품질을 위한 토큰화·동의어 정책이 필요하다.
- approximate vector index에서 강한 필터를 함께 쓰면 필터 후 남는 결과 수가 부족할 수 있다. 필터별 B-tree 인덱스, 후보 수 확대, iterative scan 또는 exact search 전환을 데이터 분포에 맞게 선택해야 한다.

#### 예상 비용

- 색인: 공지 신규·변경 시 `구조화 1회 + chunk 수만큼 임베딩`. 첨부 OCR/비전은 별도.
- 질의: 후보 1과 거의 같고, DB 내부 전문 검색이 추가된다. 선택적 reranker를 쓰면 질의당 1회가 더 든다.
- 저장: 공지 수가 아니라 전체 chunk 수에 선형. 평균 4 chunk라면 벡터 저장량도 후보 1의 약 4배다.
- 상대 등급: 구축비 **중~높음**, 질의비 **낮음~중**, 운영비 **중**.

#### 구현 난이도

중~높음. 스키마 마이그레이션, chunker, GIN/HNSW 인덱스, RRF, 평가 파이프라인이 필요하다.

#### MVP 적합성

**높음.** 일정이 매우 짧으면 D가 낫지만, 실제 사용자 공개를 목표로 하면 E의 최소 버전까지 MVP 범위에 포함할 가치가 있다. 첫 버전은 reranker 없이 RRF만 사용한다.

#### 확장 적합성

**매우 높음.** 검색량 증가, 첨부파일, 다국어, FAQ, 개인화 필터로 확장하기 쉽다. 이 문서의 최종 권고안이다.

### 후보 3: G — 검수된 FAQ 우선 + E 기반 공지 RAG

#### 데이터 흐름

```text
공지 변경 이벤트
  -> FAQ가 참조한 source_notice_id/content_hash 확인
  -> 변경·만료 FAQ 자동 비활성화 또는 재검수

사용자 질문
  -> FAQ metadata filter + 전문/벡터 검색
  -> FAQ 신뢰도 gate
       -> 통과: FAQ 답 + 최신 원문 상태 확인 + 출처 반환
       -> 실패: 후보 2(E) 공지 RAG 실행
  -> 사용자 피드백/상담 로그에서 FAQ 후보 생성
  -> 담당자 검수 후 활성화
```

#### 구성요소

- 후보 2의 모든 구성요소
- FAQ 버전 테이블과 승인 워크플로
- FAQ 임베딩·전문 검색 인덱스
- FAQ-to-notice 출처 연결
- 원문 해시·유효 기간 기반 무효화 작업
- FAQ gate와 fallback 라우터
- 질문 빈도·실패율 분석

#### 장점

- 반복 질문에 가장 빠르고 일관된 답을 준다.
- 답변 LLM 호출을 생략하거나 매우 작게 만들 수 있다.
- 검수된 표현과 안내 문구를 유지하기 쉽다.
- FAQ 미일치 시 E로 내려가므로 신규 질문도 처리한다.

#### 단점

- 오래된 FAQ가 가장 위험한 오답원이 될 수 있다.
- 의미가 비슷하지만 조건이 다른 질문을 FAQ로 잘못 라우팅할 수 있다.
- FAQ 작성·승인·무효화 운영이 필요하다.
- FAQ가 적은 초기에는 투자 대비 효과가 낮다.

#### 예상 비용

- FAQ 적중 질의: `DB 검색 + 선택적 소형 검증`, 답변 생성 비용은 0에 가깝다.
- FAQ 미적중 질의: 후보 2와 동일하고 FAQ 조회 지연만 소폭 추가된다.
- 구축·운영: 승인 UI와 무효화 로직 때문에 후보 2보다 높다.
- 상대 등급: 구축비 **높음**, 평균 질의비 **매우 낮음~낮음**, 운영비 **높음**.

#### 구현 난이도

높음. 검색보다 콘텐츠 거버넌스와 변경 동기화가 더 어렵다.

#### MVP 적합성

**중간 이하.** 초기에 이미 검수된 FAQ와 담당자가 있다면 제한적으로 적용할 수 있다. 그렇지 않으면 검색 로그가 쌓인 뒤 도입한다.

#### 확장 적합성

**매우 높음.** 트래픽과 반복 문의가 증가할수록 비용 절감 효과가 커진다.

### 후보 4: F — 지식 그래프 보조 검색 + E

#### 데이터 흐름

```text
구조화 공지
  -> 엔터티 정규화(제도/학기/대상/부서/서류/행동)
  -> 관계 추출 및 유효 기간 부여
  -> 그래프 또는 PostgreSQL 관계 테이블 저장

관계형 질문
  -> 의도 분류
  -> 그래프 조건 탐색으로 notice_id 후보 생성
  -> E의 전문/벡터 검색으로 근거 chunk 회수
  -> 관계와 원문 근거를 함께 답변
```

#### 구성요소

- 후보 2의 모든 구성요소
- 엔터티 사전과 canonical ID
- 관계 추출·검증 파이프라인
- 그래프 DB 또는 PostgreSQL edge 테이블
- 시간 구간이 있는 관계 모델
- 그래프 질의 생성기와 문서 근거 검증기

#### 장점

- 여러 공지를 가로지르는 조건·관계·선후 조건 질의에 강하다.
- 어떤 관계를 통해 답을 얻었는지 설명할 수 있다.
- 제도와 부서가 여러 공지에서 반복될 때 중복을 정규화할 수 있다.

#### 단점

- 엔터티 중복 병합과 관계 유효 기간 관리가 어렵다.
- 그래프 관계 자체가 원문 근거를 대체하지 못하므로 결국 문서 RAG가 필요하다.
- 일반적인 “기간/서류/대상” 질문에는 비용 대비 이점이 작다.

#### 예상 비용

- 공지 변경마다 엔터티·관계 재추출 및 검증 비용이 발생한다.
- 별도 그래프 DB를 선택하면 인프라와 관측 비용이 추가된다.
- 상대 등급: 구축비 **매우 높음**, 질의비 **중**, 운영비 **매우 높음**.

#### 구현 난이도

매우 높음.

#### MVP 적합성

**낮음.** 현재 MVP에는 권장하지 않는다.

#### 확장 적합성

**조건부 높음.** 관계형 질문이 실제 실패 질의의 큰 비중을 차지한다는 데이터가 있을 때 보조 계층으로 도입한다.

## 5. 색인 단위 설계

### 5.1 공지를 어떤 단위로 색인해야 하는가?

**공지 1개를 parent document로 유지하고, 검색은 parent와 의미 구간 child를 함께 사용한다.** 공지를 페이지 번호나 고정 글자 수만으로 잘게 자르면 제목·기간·대상과 본문 근거가 분리된다. 다음 계층을 권장한다.

1. `notice`: 공지의 정체성, 제목, 게시일, 원문 URL, 상태, 구조화 메타데이터
2. `notice_chunk`: 본문 또는 첨부의 의미 단락, 표, 목록, 신청 절차, 필요 서류
3. `attachment`: 파일명, MIME type, 원본 URL, 파일 해시, 추출 상태, 추출 버전
4. `chunk_citation`: 원문 본문 위치 또는 첨부 파일의 페이지·시트·섹션

공지 단위 인덱스는 제목·요약·메타데이터로 공지 후보를 찾는 데 쓰고, chunk 인덱스는 그 공지 안에서 답의 근거를 찾는 데 쓴다. 전역 chunk 검색도 병렬로 수행하되 결과는 `notice_id`로 묶어 한 공지의 유사한 chunk가 top-k를 독점하지 않게 한다.

### 5.2 공지 전체와 구조화 metadata 중 무엇을 우선 검색해야 하는가?

우선순위는 “둘 중 하나”가 아니라 다음 역할 분담이다.

- **명시된 제약**: 메타데이터를 우선한다. 예: `2026학년도`, `2학기`, `3학년`, `재학생`, `마감 전`.
- **주제와 표현**: 원문 전문 검색과 벡터 검색을 병렬 사용한다.
- **추정된 제약**: hard filter가 아니라 boost로 사용한다. 예: 질문에 연도가 없는데 시스템이 현재 학년도를 추정한 경우.
- **메타데이터 신뢰도 낮음/누락**: 해당 필드를 hard filter로 쓰지 않고 원문 검색 후보를 유지한다.

즉, metadata는 후보의 자격과 우선순위를 결정하고, 원문은 실제 답과 인용 근거를 결정한다. 답변은 메타데이터 값만 믿지 말고 선택된 원문 구간에서 다시 확인한다.

### 5.3 긴 공지는 chunking해야 하는가?

**해야 한다. 단, 모든 공지를 무조건 같은 크기로 자르지 않는다.** 권장 정책은 다음과 같다.

- 제목, 소제목, 번호 목록, 표 경계, 첨부 문서의 페이지/섹션을 먼저 보존한다.
- 짧은 공지는 한 chunk로 유지한다.
- 긴 섹션만 모델 토큰 기준 대략 400~800 tokens로 분할하고 50~100 tokens 정도 겹친다.
- 각 chunk 앞에 공지 제목, 게시일, 학년도·학기, 섹션 경로를 짧게 붙여 문맥을 보존한다.
- 표는 행을 무작정 분리하지 말고 헤더를 각 조각에 반복한다.
- 답변 단계에는 같은 공지의 인접 chunk를 합치되 전체 공지를 자동으로 넣지는 않는다.

정확한 크기는 고정 규칙이 아니라 평가 세트의 Recall@k와 답변 근거 정확도로 결정한다. 단일 임베딩 입력을 임의로 앞 2,000자에서 자르는 현재 방식은 뒤쪽의 신청 방법·서류·유의사항을 잃을 수 있으므로 개선 대상이다.

## 6. 첨부파일 처리

첨부파일은 공지 본문보다 중요한 서식·세부 일정·선발 명단을 포함할 수 있으므로 별도 문서로 관리하되 공지 parent에 연결한다.

### 수집 및 보안

1. URL, 파일명, MIME type, 크기, 파일 해시를 먼저 저장한다.
2. 허용 형식과 최대 크기를 제한하고 악성 파일 검사를 거친다.
3. 동일 해시 파일은 다시 추출·임베딩하지 않는다.
4. 원본은 변경 이력과 감사가 필요할 때만 객체 저장소에 보관하고 접근 정책을 둔다.

### 형식별 추출

- 텍스트 PDF: 레이아웃을 보존해 텍스트와 페이지 번호 추출
- 스캔 PDF/이미지: OCR; 낮은 OCR 신뢰도는 검수 또는 “원문 확인 필요” 표시
- HWP/HWPX: 안전한 변환기 또는 허용된 파서로 텍스트·표 추출
- DOCX: 제목·문단·표 구조 보존
- XLSX/CSV: 시트명과 헤더를 포함해 행 묶음으로 변환; 명단은 개인정보 정책에 따라 색인 제외 또는 마스킹
- ZIP: 기본 자동 해제 금지. 명시적 정책과 폭탄 방지 제한이 있을 때만 처리

### 색인 및 인용

- 첨부 텍스트를 공지 본문에 단순 연결한 하나의 거대 임베딩으로 만들지 않는다.
- `source_type=attachment`, `attachment_id`, 페이지/시트/행 범위를 chunk에 기록한다.
- 파일명과 서식명은 전문 검색에서 높은 가중치를 준다.
- OCR/변환 실패 파일도 파일명·URL·실패 상태는 검색 가능하게 하며, 답변에는 내용을 추정하지 않고 다운로드 확인을 안내한다.
- 원문 공지와 첨부 내용이 충돌하면 최신 버전 및 명시적 정정 공지를 우선하고 충돌 사실을 사용자에게 알린다.

## 7. 날짜·학년도·학기 추출

### 결론: 규칙 기반을 기준으로 하고 LLM은 보완에 사용한다

날짜와 학기는 정렬·필터·만료 판정에 직접 쓰이므로 재현 가능하고 테스트 가능한 규칙이 기준이어야 한다.

#### 규칙 기반이 담당할 것

- `YYYY-MM-DD`, `YYYY.MM.DD`, `M월 D일`, 시간, 요일
- `2026학년도`, `1학기`, `2학기`, `하계/동계 계절학기`
- `~까지`, `부터`, `연장`, `마감`, `상시`와 같은 기간 표지
- 게시일을 기준으로 연도가 생략된 날짜의 후보 연도 생성
- KST(`Asia/Seoul`) 기준 시간 정규화
- 시작/종료 역전, 잘못된 날짜, 요일 불일치 검증

#### LLM이 담당할 것

- 여러 날짜 중 신청 기간, 행사 기간, 서류 발급일을 역할별로 분류
- “개강 후 2주 이내”, “성적 확정 다음 날” 같은 상대 표현 해석 보조
- 표와 문맥에 흩어진 날짜 연결
- 규칙 결과가 여러 개인 경우 가장 가능성 높은 구조 제안

#### 저장 원칙

- 원문 표현(`raw_text`)과 정규화 값(`normalized_at`)을 모두 저장한다.
- `extraction_method`, `confidence`, `reference_date`, `timezone`을 기록한다.
- 연도가 생략되었거나 상대 날짜인 경우 `inferred=true`로 표시한다.
- 규칙과 LLM이 충돌하면 자동 확정하지 말고 신뢰도를 낮추거나 검수 큐로 보낸다.
- 학년도와 달력 연도를 분리한다. 2026학년도 동계 계절학기가 2027년 1월에 실행될 수 있다.

질의의 “이번 학기”도 단순히 1~7월=1학기, 나머지=2학기로 고정하지 않는다. 대학 학사 일정 테이블을 기준으로 현재 학기와 계절학기를 계산하고, 경계 기간에는 사용자에게 기준 학기를 명시한다.

## 8. PostgreSQL 전문 검색과 pgvector를 함께 사용할 가치

**가치가 있다. E를 운영 기본안으로 권장한다.** 두 검색은 오류 유형이 다르다.

| 질의 유형 | 전문 검색 | 벡터 검색 |
|---|---:|---:|
| “분할납부 신청서” 정확 명칭 | 강함 | 중간 |
| “등록금 나눠 내고 싶어” 구어체 | 약함~중간 | 강함 |
| 학수번호·부서명·서식번호 | 매우 강함 | 약함 |
| 긴 문장의 의미·동의어 | 중간 | 강함 |
| 왜 검색되었는지 하이라이트 | 강함 | 약함 |

### 권장 결합 방식

초기에는 점수 정규화가 필요 없는 RRF를 사용한다.

```text
rrf_score(d) = Σ 1 / (k + rank_i(d))
```

- 전문 검색과 벡터 검색에서 각각 top 30~50을 구한다.
- `k`는 60을 시작값으로 두되 평가로 보정한다.
- 결합 후 명시 메타데이터 일치, 상태, 최신성, 추출 신뢰도로 재정렬한다.
- 상위 결과가 비슷할 때만 cross-encoder 또는 소형 LLM reranker를 선택적으로 호출한다.

### PostgreSQL 구현 방향

- 제목에는 A, 구조화 키워드·동의어·파일명에는 B, 본문에는 C, 첨부 추출문에는 C 또는 D 가중치
- `tsvector` 생성 열과 GIN 인덱스 사용
- 사용자 입력에는 문법 오류에 안전한 `websearch_to_tsquery` 또는 정규화된 `plainto_tsquery` 사용
- 벡터는 데이터가 작을 때 exact search로 시작하고, 규모·지연 측정 후 HNSW 도입
- 필터 컬럼에는 B-tree/적절한 배열 인덱스를 별도로 둔다
- 한국어 Recall@k가 부족할 때만 형태소 분석 확장을 도입한다

PostgreSQL 문서는 `ts_rank_cd`가 cover density 기반 관련도 점수를 계산한다고 설명하며, pgvector 문서는 전문 검색과의 하이브리드 검색 및 RRF/cross-encoder 결합을 명시적으로 제안한다. 따라서 별도 검색 클러스터 없이도 이 프로젝트 규모에서 실용적인 하이브리드 검색을 구성할 수 있다.

## 9. 검색 실패 판정

검색 실패는 “결과가 0개”뿐 아니라 “결과는 있지만 답을 뒷받침하지 못함”을 포함한다. 다음 2단계 판정을 권장한다.

### 9.1 검색 단계 실패

아래 중 하나면 필터 완화 재시도를 수행한다.

- hard filter 후 후보가 0개
- top 후보가 최소 결합 점수 미만
- top 1과 top 2가 모두 낮고 점수 차도 작아 구분이 안 됨
- 질문의 명시 필수 조건(연도·학기·학년·상태)을 만족한 후보가 없음
- 모든 후보가 낮은 신뢰도의 추출 필드에만 의존

재시도 순서는 다음과 같다.

1. 추정된 조건만 hard filter에서 boost로 완화
2. 동의어·띄어쓰기·약어 확장
3. 후보 수 확대 및 전문/벡터 중 한쪽의 단독 후보 포함
4. current 질의라면 만료 공지를 제외한 채 기간 미상 공지까지 확대
5. 그래도 실패하면 “관련 데이터가 없다”를 반환하고 담당 부서/검색 링크 안내

사용자가 명시한 `2025학년도`나 `3학년` 조건을 임의로 완화해서는 안 된다. 완화가 필요하면 조건을 충족하는 자료가 없음을 먼저 알리고 별도 섹션에 근접 결과를 제시한다.

### 9.2 답변 근거 실패

검색 결과가 있어도 다음 조건이면 확정 답변을 만들지 않는다.

- 신청 기간 질문인데 선택된 chunk에 기간 근거가 없음
- 학년 자격 질문인데 대상 필드와 원문 모두 학년을 확인할 수 없음
- 서로 다른 공지의 날짜가 충돌하고 최신 정정 관계가 불명확
- top 문서가 만료되었는데 사용자는 현재 신청 가능 여부를 물음
- OCR/LLM 추출 신뢰도가 기준 이하이고 원문으로 교차 확인되지 않음

이 경우 답변 상태를 `no_result`, `insufficient_evidence`, `conflicting_evidence`, `stale_only` 등으로 구분해 관측한다. 사용자에게는 무엇을 확인하지 못했는지 구체적으로 말하고 출처 없는 추측을 하지 않는다.

### 9.3 임계값 설정

현재 코드의 고정 점수 `0.22`는 임시값으로만 취급한다. 검색 방식과 점수 분포가 바뀌면 의미가 달라진다. 최소 100~300개의 실제형 질문과 정답 공지·근거 구간을 만든 뒤 다음 지표로 보정한다.

- 공지 Recall@5, MRR 또는 nDCG@5
- 근거 chunk Recall@10
- 명시 조건 위반율
- 만료 공지 오답률
- 무응답 정밀도: 실패라고 판정한 질의 중 실제 실패 비율
- 답변 citation precision 및 담당자 검수 통과율

초기 운영에서는 오답보다 정직한 무응답을 선호해 임계값을 보수적으로 두고, 실패 로그를 검수해 점진적으로 완화한다.

## 10. 종료·보관 공지의 우선순위

`is_archived`와 `notice_status=expired`는 서로 다른 의미로 관리한다.

- `expired`: 신청/행사 기간이 지났지만 원문은 여전히 유효한 과거 기록
- `archived`: 원문이 삭제·교체되었거나 정상 수집 범위에서 사라진 상태
- `superseded`: 후속 또는 정정 공지가 명시적으로 대체한 상태(추가 권장)

### 질의 의도별 처리

| 질의 의도 | active/upcoming | unknown/always | expired | archived/superseded |
|---|---:|---:|---:|---:|
| 현재 신청·가능 여부 | 강한 가점 | 약한 가점/검증 | 강한 감점 | 기본 제외 |
| 특정 과거 연도·학기 | 조건 일치 점수 | 조건 일치 점수 | 감점 없음 | 필요 시 포함 |
| 절차·서식 일반 안내 | 최신 공지 우선 | 포함 | 약한 감점 | 기본 제외 |
| 변경 이력·과거 공지 요청 | 포함 | 포함 | 포함 | 명시적으로 포함 |

### 상태 계산

- 상태를 색인 시 한 번 정하고 끝내지 않는다.
- `application_start/end`, `event_start/end`, 명시적 `상시`, 게시 중단, 정정 관계를 바탕으로 KST 기준 상태를 주기적으로 재계산하거나 질의 시 계산한다.
- `expired`를 절대 필터로 제거하지 않는다. current 질의에서 예를 들어 `-0.2` 같은 감점 또는 별도 순위 그룹을 쓰고, 정확한 값은 평가로 정한다.
- 현재 공지가 하나도 없고 만료 공지만 있다면 “현재 유효한 공지는 찾지 못했으며, 아래는 과거 공지”라고 분리해 제시한다.
- 정정·대체 공지가 있으면 이전 공지보다 후속 공지를 우선하고 관계를 함께 표시한다.

현재 구현은 `is_archived=False`를 항상 적용하므로 과거 이력 요청도 보관 공지를 찾지 못한다. `time_scope=historical` 또는 명시적 과거 연도 질의에서는 보관 공지를 포함할 수 있도록 정책을 분리해야 한다.

## 11. 권장 스키마 변경

기존 `notices`, `notice_metadata`, `notice_embeddings`를 유지하면서 다음을 추가한다.

```text
notices
  + superseded_by_notice_id
  + source_updated_at
  + status_computed_at

notice_metadata
  + academic_terms[] 또는 별도 notice_terms
  + extraction_method / confidence_by_field
  + raw_date_expressions
  + target_grades용 PostgreSQL array 또는 연결 테이블

notice_chunks
  id, notice_id, attachment_id?, source_type, section_path,
  chunk_order, chunk_text, token_count, content_hash,
  search_document tsvector, embedding vector,
  page_start?, page_end?, extraction_confidence

attachments
  id, notice_id, filename, source_url, mime_type, byte_size,
  file_hash, extraction_status, extractor_version, extracted_at

notice_relations
  from_notice_id, to_notice_id, relation_type, confidence
  # corrects / supersedes / related

search_events
  query_hash, parsed_filters, candidate_ids, ranks, scores,
  failure_reason, selected_notice_ids, latency_ms, feedback
```

학년 배열을 JSON으로만 유지하면 효율적인 포함/겹침 검색과 인덱싱이 불편하다. PostgreSQL 운영 환경에서는 `smallint[]` + GIN 또는 정규화된 연결 테이블을 권장한다. 학년도·학기는 하나의 공지에 여러 값이 나올 수 있으므로 현재 단일 정수 필드만으로 충분한지 실제 데이터로 검증한다.

## 12. 권장 검색 정책

### 12.1 hard filter

- 사용자가 명시한 학년도·학기
- 사용자가 명시한 학년·학생 상태·캠퍼스(단, 메타데이터 신뢰도가 충분할 때)
- 현재 가능 공지만 요구한 경우 `active/upcoming/always`; `unknown`은 보조 후보
- 개인정보·권한·공개 범위

### 12.2 soft boost

- 질의 분석기가 추정한 카테고리·하위 카테고리
- 현재 시각으로 추정한 학년도·학기
- 부서, 유사 학과명, 동의어
- 최신 게시일
- active/upcoming 상태

카테고리는 추출 오류 가능성이 있어 기본적으로 soft boost가 안전하다. 후보가 많을 때만 confidence가 높은 카테고리를 hard filter로 승격한다.

### 12.3 권장 초기 랭킹

1. 전문 검색 top 40, vector top 40 생성
2. RRF로 결합
3. 명시 조건 일치와 상태를 반영해 재정렬
4. 공지별 최대 2 chunk만 유지하여 다양성 확보
5. top 10에 대해 근거 요건 검사
6. top 3 공지, 총 4~8 chunk를 답변 컨텍스트로 사용

랭킹 가중치와 후보 수는 운영 데이터 없이 확정하지 않는다. 위 수치는 시작점이다.

### 1단계: 현재 D 방식 보강

- 질문의 `grade`, `student_status`, `department`, `time_scope`를 실제 검색 조건에 반영

- 만료 상태를 질의 시 재계산하고 의도별 감점
- hard filter 0건 시 안전한 완화 재시도
- 검색 이벤트와 실패 이유 기록
- 평가 질문 세트 구축

### 2단계: E 최소 버전

- `notice_chunks`와 첨부 테이블 마이그레이션
- 구조 기반 chunker 구현
- `tsvector`/GIN 전문 검색 추가
- 전문 top-k와 vector top-k를 RRF로 결합
- 공지·chunk 중복 제거와 근거 인용
- 변경 chunk만 재임베딩

### 3단계: 운영 품질

- 첨부 PDF/HWPX/DOCX/XLSX 추출 워커

### 4단계: 트래픽 기반 확장

- 반복·고신뢰 질문만 FAQ로 승격
- 공지 해시 기반 FAQ 자동 무효화
- 실제 실패 유형이 입증할 때만 지식 그래프 보조 검색 검토

## 14. 최종 선택

| 시점 | 선택 | 이유 |
|---|---|---|
| 지금 즉시 | D 보강 | 현재 코드와 스키마를 활용해 가장 빨리 조건 정확도와 실패 처리를 개선 |
| 실제 MVP 공개 전/직후 | E 최소 버전 | 정확어·구어체·긴 공지를 함께 처리하며 근거 구간 제시 가능 |
| 반복 문의 데이터 축적 후 | G + E | 검수된 FAQ로 지연·비용 절감, 미일치 질의는 E로 안전하게 fallback |
| 관계형 질의 수요 검증 후 | F + E | 그래프가 실제로 해결하는 실패 유형이 확인될 때만 투자 |

따라서 목표 아키텍처는 **E**, 단계적 구현 순서는 **D → E → 선택적 G**다. F는 대체재가 아니라 특정 관계 질의를 위한 장기 보조 계층이다. A는 데모 외에는 제외하고, B와 C는 각각 E를 구성하는 검색 채널 또는 장애 fallback으로 사용한다.
